| Слой | Лист(ы) | Целевой источник | Примечание |
|---|---|---|---|
| RAW (регулярный поток) | `RAW_WB_ORDERS`, `RAW_WB_SALES_RETURNS`, `RAW_WB_FINANCE`, `RAW_WB_ADS`, `RAW_WB_STOCKS`, `RAW_WB_STORAGE` | WB API | Единственный производственный источник регулярной аналитики |
| RAW (сверка/история/fallback) | те же листы, помеченные `load_id` Excel/Drive | Excel/Drive | Только вне регулярного потока, с явной пометкой источника |
| CLEAN | `CLEAN_WB_DAILY` | производный от RAW | Источник наследуется от RAW; в целевой модели ручных формул поверх RAW быть не должно |
| ЮНИТ / P&L | `UNIT_SKU_DAILY`, `PNL_TOTAL` | производный от CLEAN/RAW | Факт берётся из API-данных в RAW; себестоимость — из справочников |
| DASHBOARD | `DASHBOARD_WB` и витрины | производный | Отображение, без собственного источника факта |

> Себестоимость в эту схему как «WB API → RAW» **не входит**: она поступает из внутренних справочников (§8 ниже) и подмешивается на уровне ЮНИТ / P&L.

## 5. Отдельный раздел по рекламе

- Реклама **не считается точным SKU-расходом**, пока фактически не подтверждена надёжная связка `date + nmId + advertId + spend` в данных WB API.
- До подтверждения этой связки рекламные расходы учитываются одним из способов:
  - **кабинетный расход** — общий расход на рекламу по кабинету (строка верхнего блока, например «WB Продвижение» / «Расход на рекламу»), без разнесения по SKU; либо
  - **распределяемая оценка** — разнесение по SKU только по **утверждённому правилу** и **только с явной пометкой «оценка»** (не «факт»).
- Целевая структура `RAW_WB_ADS` должна позволять хранить поля `advert_id`, `stat_date`, `wb_nm_id`, `spend`; фактическое наличие и корректность этой связки в реальных API-данных требует проверки (`TBD`).
- Обязательная сверка при проверке: `sum(RAW_WB_ADS.spend)` за период ↔ удержание «WB Продвижение» в `RAW_WB_FINANCE`. Расхождение (`delta`) фиксируется и классифицируется; до согласования `delta` реклама по SKU остаётся оценкой.
- Запрет: показывать рекламу по SKU как факт при недоказанной связке — нарушение правила из `CLAUDE.md` и `stabilization_plan.md`.

---

## 6. Отдельный раздел по неотнесённым расходам

Расход, который **нельзя надёжно отнести к конкретному SKU**, учитывается строго одним из утверждённых способов:

- **кабинетный расход** — относится на кабинет целиком, не разносится по SKU (примеры из текущей модели: «Возмещение ПВЗ», «Транзитные поставки WB», часть удержаний и приёмки);
- **отдельный блок** — выделяется в самостоятельную строку/блок отчёта без искажения SKU-экономики;
- **распределение по утверждённому правилу** — разносится по SKU **только** по заранее зафиксированному правилу и **только** с пометкой «оценка».

Запреты:
- **Запрещено ручное распределение без утверждённого правила.** Способ учёта каждого неотнесённого расхода фиксируется отдельным решением с записью в `CHANGELOG.md`.
- Запрещено выдавать оценку за факт. Любое распределение по правилу маркируется как «оценка».
- Запрещён двойной учёт: если расход уже учтён кабинетным блоком, он не разносится повторно по SKU.

## 7. Отдельный раздел по внутренним справочникам

Себестоимость и состав товаров **не приходят из WB API** — это нормально и соответствует модели API-first. Проверяется не наличие в API, а **полнота и корректность внутреннего справочника**.

| Справочник | Назначение | Ключи / связь | Что проверять |
|---|---|---|---|
| `SKU_MASTER` | Главный справочник товаров; маппинг `wb_nm_id` / `barcode` / `wb_vendor_code` → `internal_sku` | `internal_sku` ↔ `wb_nm_id` | Полнота активных SKU; однозначность маппинга; `sku_match_status` |
| `COST_HISTORY` | История себестоимости по SKU/дате | `internal_sku` + дата | Наличие COGS на нужные даты; отсутствие пропусков по активным SKU |
| `BUNDLES` | Состав наборов (комплект → компоненты) | набор `internal_sku` → состав | Корректная декомпозиция COGS набора по компонентам |

- COGS подмешивается на уровне **ЮНИТ / P&L**, а не в RAW из API.
- Контрольные суммы по `SKU_MASTER` и `COST_HISTORY` обязательны (по `stabilization_plan.md`).
- Себестоимость **не искать в WB API** ни при каких условиях.

## 8. Критерии готовности API-only блока

Блок считается готовым к работе в режиме API-only только при выполнении **всех** условий (по `data_source_policy.md` §9):

1. Блок доступен через WB API в нужной детализации (по дате и nmId/advertId, где применимо) — подтверждено фактически, а не предположительно.
2. Точный метод/версия/параметры API зафиксированы (не `TBD`).
3. Целевой RAW-лист и маппинг полей подтверждены; привязка к SKU через `internal_sku` работает (контроль `sku_match_status`).
4. Данные API за контрольный период сверены; расхождение API vs Excel/Drive разобрано и классифицировано (ошибка загрузки / разница методологии / округление / непокрытый блок).
5. Загрузка идемпотентна, без дублей (контроль через `IMPORT_LOG_*`, дедуп по `row_hash`).
6. Для рекламы — подтверждена связка `date + nmId + advertId + spend`, иначе блок остаётся «кабинет/оценка» (§5).
7. Способ учёта неотнесённой части зафиксирован по утверждённому правилу (§6).
8. Решение и контрольные значения записаны в `CHANGELOG.md`; соответствующий Excel/Drive-модуль помечен legacy.

До выполнения всех условий блок остаётся в смешанном режиме, Excel/Drive сохраняет роль сверки/fallback, а статус в карте — `TBD / требует проверки`.

## 9. Что не делать

| Не делать | Почему |
|---|---|
| Объявлять любой блок «покрыт API» до фактической проверки | Покрытие не подтверждено; статус по умолчанию — `TBD` |

| Искать себестоимость в WB API | COGS — из `SKU_MASTER` / `COST_HISTORY` / `BUNDLES` |
| Показывать рекламу по SKU как факт без связки `date + nmId + advertId + spend` | Нарушает правило `CLAUDE.md` / `stabilization_plan.md`; до подтверждения — оценка/кабинет |

| Закрывать пробелы покрытия Excel/Drive в обход политики | Непокрытый блок фиксируется как `TBD`, а не подменяется legacy-источником |

## 10. Следующий безопасный шаг после заполнения карты

1. Провести **фактическую инвентаризацию** WB API по блокам §4: подтвердить точный метод/версию, детализацию и наличие ключей (`srid`, `nmId`, `advertId`); заменить `TBD` на «доступно / частично / недоступно» рядом со статусами. Только проверка, без изменения кода.
2. Отдельно проверить **связку рекламы** `date + nmId + advertId + spend` в `RAW_WB_ADS` и сверить `sum(spend)` ↔ удержание «WB Продвижение» в `RAW_WB_FINANCE`; зафиксировать `delta`.
3. Проверить **полноту внутренних справочников** (`SKU_MASTER`, `COST_HISTORY`, `BUNDLES`) на активных SKU — для COGS на уровне ЮНИТ / P&L.
4. Для покрытых блоков снять API-данные за контрольный период и сверить; результат сверки — основа будущего `docs/control_values.md` (фиксируется только после подтверждения по API).
5. Записать итоги (покрытие, пробелы `TBD`, классификацию расхождений, решения по неотнесённым расходам) в `CHANGELOG.md`. Excel/Drive-модули пометить legacy, не удаляя.
6. После согласования карты использовать `docs/wb_api_coverage_map.md` как рабочую основу для проверки API-покрытия и будущего `docs/control_values.md`.
