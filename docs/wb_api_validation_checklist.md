# WB API Validation Checklist — EVETIS WB Analytics

> Чек-лист фактической проверки WB API для проекта **EVETIS WB Analytics**.
> Целевая модель проекта: **API-first / API-only**.
> Поток данных: `WB API → RAW → CLEAN → ЮНИТ / P&L → DASHBOARD`.

Документ не меняет код, не создаёт новые функции, не меняет структуру Google Sheets и не утверждает покрытие WB API до фактической проверки.

Все непроверенные блоки имеют статус `TBD / требует проверки`.

---

## 1. Цель документа

Подготовить пошаговую проверку каждого блока данных из `docs/wb_api_coverage_map.md` перед разработкой кода.

Для каждого блока нужно фактически подтвердить:

* точный WB API раздел / endpoint / метод;
* категорию токена и права доступа;
* обязательные и дополнительные параметры запроса;
* период истории, который реально можно получить;
* поля, которые возвращает API;
* целевой RAW-лист;
* уровень детализации: кабинет / операция / дата / SKU / `nmId` / `advertId`;
* ключи связи;
* возможность надёжной привязки к SKU;
* место показателя в P&L;
* статус: `доступно` / `частично` / `недоступно` / `требует проверки`;
* риски, расхождения и ограничения.

---

## 2. Жёсткие правила проверки

### 2.1. Что разрешено

* Проверять документацию WB API.
* Выполнять тестовые API-запросы на ограниченном периоде.
* Сохранять результаты проверки в отдельные заметки / временные файлы вне боевого кода.
* Сравнивать API-данные с Excel/Drive только как со сверкой.
* Фиксировать расхождения как отдельный результат проверки.

### 2.2. Что запрещено

| Запрещено                                                              | Почему                                                           |
| ---------------------------------------------------------------------- | ---------------------------------------------------------------- |
| Менять код до завершения проверки                                      | Сейчас этап аудита покрытия, не разработки                       |
| Создавать новые функции Apps Script                                    | Сначала подтверждаем источники и поля                            |
| Объявлять API-блок покрытым без реального запроса                      | Документация и гипотеза не равны факту                           |
| Подменять непокрытый API-блок Excel/Drive                              | Нарушает API-first / API-only политику                           |
| Фиксировать Excel/Drive как окончательный эталон                       | Контрольные значения должны подтверждаться по API                |
| Искать себестоимость в WB API                                          | COGS берётся из `SKU_MASTER`, `COST_HISTORY`, `BUNDLES`          |
| Считать рекламу SKU-фактом без связки `date + nmId + advertId + spend` | Без этой связки реклама остаётся кабинетным расходом или оценкой |
| Распределять неотнесённые расходы вручную без правила                  | Это искажает SKU-экономику                                       |

---

## 3. Статусы проверки

| Статус                   | Значение                                                                                      |
| ------------------------ | --------------------------------------------------------------------------------------------- |
| `TBD / требует проверки` | Блок ещё не проверен фактически                                                               |
| `доступно`               | API даёт нужные данные в нужной детализации                                                   |
| `частично`               | API даёт данные, но не хватает полей, детализации или периода                                 |
| `недоступно`             | API не покрывает нужный показатель                                                            |
| `blocked`                | Проверка невозможна сейчас: нет токена, нет прав, ошибка доступа, лимит, техническая проблема |
| `legacy сверка`          | Excel/Drive допустим только для сверки, аудита или старых периодов                            |

## 6. Проверка блока «Заказы»

### 6.1. Что нужно подтвердить

* Точный endpoint и метод для заказов.
* Категорию токена.
* Параметры `dateFrom`, `flag` или их актуальные аналоги.
* Доступную глубину истории.
* Обновляемость данных.
* Есть ли `srid`.
* Есть ли `nmId`, `barcode`, `supplierArticle`.
* Как обрабатывать отменённые заказы.
* Можно ли использовать данные как операционный показатель, но не как финансовый факт.

### 6.2. Проверочные вопросы

* Возвращает ли API одну строку на один товар в заказе?
* Есть ли стабильный ключ операции?
* Можно ли дедуплицировать по `srid`?
* Что происходит с отменёнными заказами?
* Отличаются ли суммы заказа от фактической реализации в фин. отчёте?
* За какой период реально доступны данные для EVETIS?

### 6.3. Решение после проверки

* Если данные полные: использовать в воронке и операционной аналитике.
* Если суммы отличаются от фин. отчёта: не использовать как P&L-факт.
* Если история ограничена: зафиксировать ограничение в карте покрытия.

---

## 7. Проверка блока «Продажи»

### 7.1. Что нужно подтвердить

* Точный endpoint и метод для продаж.
* Как API разделяет продажи и возвраты.
* Есть ли `saleID`, `srid`, `nmId`, `barcode`, `supplierArticle`.
* Какие поля суммы возвращаются.
* Какие поля предварительные, а какие пригодны для финального P&L.
* Нужно ли считать продажи из этого блока операционным показателем, а финальный факт брать из финансового отчёта.

### 7.2. Проверочные вопросы

* Одна строка = одна продажа / возврат / товар?
* Есть ли отдельный признак возврата?
* Есть ли отрицательные количества или отдельные типы операций?
* Можно ли связать продажу с заказом по `srid`?
* Совпадает ли сумма продаж с финансовым отчётом за тот же период?
* Какие расхождения возникают и почему?

### 7.3. Решение после проверки

* Если данные предварительные: использовать для операционного мониторинга.
* Для P&L подтверждать финальный факт через финансовый отчёт.
* Возвраты фиксировать отдельно, не смешивать с продажами без признака операции.

---

## 8. Проверка блока «Возвраты»

### 8.1. Что нужно подтвердить

* Где возвраты отражаются точнее: в Sales API, финансовом отчёте или обоих источниках.
* Какой признак возврата использовать.
* Как возвраты влияют на выручку, комиссию, логистику и к перечислению.
* Можно ли связать возврат с исходной продажей.

### 8.2. Проверочные вопросы

* Есть ли `srid` у возврата?
* Есть ли исходный `saleID`?
* Возврат приходит отдельной строкой или отрицательной суммой?
* На какую дату относить возврат: дата продажи, дата возврата, дата отчёта?
* Как возвраты отражаются в финансовом отчёте реализации?
* Есть ли расхождение между Sales API и Finance API?

### 8.3. Решение после проверки

* Для операционной аналитики можно использовать Sales API, если признаки возврата надёжны.
* Для P&L приоритет должен быть у финансового отчёта, если он даёт финальный факт.
* Если привязка к SKU неполная — часть возвратов учитывать как кабинетный блок или `TBD`.

---

## 9. Проверка блока «Остатки»

### 9.1. Что нужно подтвердить

* Какой endpoint использовать как целевой: текущий остаточный отчёт, warehouse remains или другой актуальный метод.
* Не является ли выбранный endpoint deprecated.
* Есть ли детализация по складу.
* Есть ли `nmId`, `barcode`, `supplierArticle`.
* Какие поля доступны: доступно, в пути к клиенту, в пути от клиента, полный остаток, склад.
* Можно ли строить snapshot на дату.

### 9.2. Проверочные вопросы

* API отдаёт текущий snapshot или историю изменений?
* Можно ли получить остатки на конкретную прошлую дату?
* Как часто можно обновлять остатки?
* Как обрабатываются товары в пути?
* Есть ли склады WB и склады продавца отдельно?
* Все активные SKU EVETIS присутствуют?

### 9.3. Решение после проверки

* Остатки не являются P&L-фактом.
* Использовать для оборачиваемости, планирования поставок и контроля наличия.
* Если endpoint deprecated — не строить на нём долгосрочную архитектуру без альтернативы.

---

## 10. Проверка блока «Финансовый отчёт реализации»

### 10.1. Что нужно подтвердить

* Новый актуальный endpoint финансового отчёта.
* Не использовать deprecated endpoint как целевой, если WB уже дал замену.
* Параметры периода.
* Пагинацию / `rrdId` / reportId.
* Возможность получать отчёт по дням или неделям.
* Полный список возвращаемых полей.
* Наличие `nmId`, `sku`, `vendorCode`, `srid`.
* Наличие полей реализации, комиссии, эквайринга, логистики, удержаний, к перечислению.
* Возможность построить `RAW_WB_FINANCE` без Excel/Drive.

### 10.2. Проверочные вопросы

* Какой метод является актуальным после отказа от deprecated endpoint?
* Есть ли детализация по операции?
* Есть ли детализация по `nmId`?
* Какие поля отвечают за:

  * реализацию;
  * комиссию;
  * эквайринг;
  * логистику;
  * хранение;
  * удержания;
  * к перечислению;
  * тип операции;
  * дату операции;
  * дату отчёта.
* Как API обрабатывает большие отчёты?
* Нужно ли делать несколько запросов до полного получения отчёта?
* Какие поля совпадают с Excel-отчётом WB, а какие отличаются по методологии?

### 10.3. Решение после проверки

* Если финансовый отчёт даёт полный операционный уровень, он становится главным API-источником финансового факта.
* Если часть расходов не имеет SKU-привязки, не разносить вручную без правила.
* Все расхождения с Excel/Drive фиксировать как аудит, а не как замену API.

---

## 11. Проверка блока «Логистика»

### 11.1. Что нужно подтвердить

* Где логистика отражается: в финансовом отчёте, отдельном отчёте или обоих источниках.
* Какие поля отвечают за доставку к клиенту, возвратную логистику, платную логистику и корректировки.
* Есть ли `nmId`.
* Можно ли отнести логистику к SKU как факт.

### 11.2. Проверочные вопросы

* Каждая логистическая операция имеет `nmId`?
* Есть ли операции без SKU?
* Как логистика отражается при возврате?
* Есть ли дубли между финансовым отчётом и отдельными отчётами?
* Сходится ли сумма логистики с кабинетом WB и Excel-сверкой?
* На какую дату относить логистику: дата операции, дата продажи, дата отчёта?

### 11.3. Решение после проверки

* Если есть `nmId` и операция — можно относить к SKU.
* Если нет `nmId` — кабинетный расход или распределение только по утверждённому правилу.
* Не допускать двойного учёта логистики.

---

## 12. Проверка блока «Хранение»

### 12.1. Что нужно подтвердить

* Есть ли отдельный API-отчёт платного хранения.
* Максимальный период одного запроса.
* Механика task-based отчёта: создать задачу → проверить статус → скачать результат.
* Есть ли детализация по `nmId`, складу, дате.
* Совпадает ли сумма хранения с финансовым отчётом.

### 12.2. Проверочные вопросы

* Отчёт хранения даёт строки по SKU или только агрегат?
* Есть ли `nmId` / barcode / vendorCode?
* Есть ли дата начисления?
* Есть ли склад?
* Как учитывать хранение в P&L: по SKU-факту, кабинетом или оценкой?
* Не дублируется ли хранение с удержаниями в финансовом отчёте?

### 12.3. Решение после проверки

* Если хранение даёт `date + nmId + amount`, можно использовать как SKU-факт.
* Если только агрегат — учитывать кабинетным расходом.
* Если есть разница между storage report и finance report — зафиксировать `delta` и причину.

---

## 13. Проверка блока «Удержания»

### 13.1. Что нужно подтвердить

* Какие удержания приходят в финансовом отчёте.
* Какие удержания приходят в отдельных retention reports.
* Есть ли детализация по `nmId`.
* Какие удержания являются SKU-расходом, а какие кабинетным расходом.
* Как классифицировать удержания по типам.

### 13.2. Проверочные вопросы

* Есть ли тип удержания / описание операции?
* Есть ли сумма удержания?
* Есть ли дата удержания?
* Есть ли `nmId`?
* Есть ли операции без товара?
* Какие удержания нельзя разнести к SKU?
* Совпадает ли сумма удержаний с кабинетом WB и Excel-сверкой?

### 13.3. Решение после проверки

* Удержания с `nmId` можно рассматривать как SKU-факт после проверки.
* Удержания без `nmId` — кабинетный расход или отдельный блок.
* Любое распределение по SKU только по утверждённому правилу и с пометкой «оценка».

---

## 14. Проверка блока «Платная приёмка»

### 14.1. Что нужно подтвердить

* Есть ли отдельный API-отчёт расходов на приёмку.
* Максимальный период одного запроса.
* Механика task-based отчёта.
* Есть ли `nmId`, `incomeId`, дата, сумма.
* Можно ли связать приёмку с поставкой и SKU.

### 14.2. Проверочные вопросы

* Отчёт отдаёт расходы по SKU или по поставке?
* Есть ли `incomeId`?
* Есть ли `nmId`?
* Есть ли дата приёмки?
* Как соотнести приёмку с конкретным товаром?
* Не дублируется ли платная приёмка в финансовом отчёте?

### 14.3. Решение после проверки

* Если есть `nmId` и сумма — можно рассматривать как SKU-факт.
* Если только поставка / кабинет — учитывать отдельно или распределять по утверждённому правилу.
* Без правила не разносить на SKU.

## 15. Проверка блока «Реклама»

### 15.1. Что нужно подтвердить

* Какие методы Promotion / Advert API доступны для EVETIS.
* Какие типы кампаний используются: WB Продвижение, поиск, каталог, автоматические кампании, media.
* Есть ли статистика по `advertId`.
* Есть ли статистика по `date`.
* Есть ли статистика по `nmId`.
* Какие поля расходов доступны: `sum`, `spend` или другое название.
* Можно ли получить полную связку `date + nmId + advertId + spend`.

### 15.2. Проверочные вопросы

* API возвращает расходы на уровне кампании или товара?
* Есть ли вложенный массив `nms`?
* Есть ли в `nms` поле `nmId`?
* Есть ли в той же структуре дата?
* Есть ли в той же структуре расход?
* Поле расхода называется `sum`, `spend` или иначе?
* Можно ли однозначно привести расход к `spend` в `RAW_WB_ADS`?
* Есть ли кампании, где нет SKU-детализации?
* Есть ли расхождение между рекламной статистикой и удержанием рекламы в финансовом отчёте?

### 15.3. Обязательная проверка рекламы

Реклама может быть признана SKU-фактом только если фактически подтверждена связка:

```text
date + nmId + advertId + spend
```

Где:

* `date` — дата рекламного расхода;
* `nmId` — артикул WB;
* `advertId` — ID рекламной кампании;
* `spend` — расход, нормализованное поле на основе фактического поля WB API.

Если хотя бы одного элемента нет, рекламный расход нельзя показывать как SKU-факт.

### 15.4. Решение после проверки

* Если связка подтверждена: можно строить SKU-рекламу как факт.
* Если есть только `date + advertId + spend`: реклама остаётся расходом кампании / кабинета.
* Если есть `nmId`, но нет spend на уровне `nmId`: SKU-расход не подтверждён.
* Если есть spend по SKU, но нет сверки с финансовым удержанием: до сверки показывать как предварительно.
* Любой `delta` между Ads API и Finance API фиксировать отдельно.

## 16. Проверка блока «Рекламные расходы»

### 16.1. Что нужно подтвердить

* Где WB фиксирует фактическое списание денег за рекламу.
* Где WB показывает рекламную статистику.
* Как связать статистику рекламы с финансовым списанием.
* Можно ли сверить `sum(RAW_WB_ADS.spend)` с удержанием «WB Продвижение» в `RAW_WB_FINANCE`.

### 16.2. Проверочные вопросы

* В рекламном API сумма — это начисление, списание или статистическая сумма?
* В финансовом отчёте реклама приходит как удержание?
* Есть ли НДС / корректировки / бонусные списания?
* Совпадает ли период рекламы с периодом финансового отчёта?
* Есть ли задержка между показами рекламы и удержанием денег?
* Какие типы кампаний попадают в статистику, но не попадают в фин. отчёт, и наоборот?

### 16.3. Решение после проверки

* Если Ads API и Finance API сходятся — можно использовать Ads API как детализацию, Finance API как контроль.
* Если не сходятся — реклама остаётся кабинетным расходом или оценкой до классификации `delta`.
* Не разносить рекламный расход по SKU без доказанной связки и сверки.

## 17. Проверка внутренних справочников

Себестоимость не приходит из WB API. Для API-first модели это нормально: API даёт факт WB, внутренние справочники дают COGS и состав товаров.

### 17.1. `SKU_MASTER`

Проверить:

* все активные SKU EVETIS есть в справочнике;
* у каждого SKU заполнены:

  * `internal_sku`;
  * `wb_nm_id`;
  * barcode;
  * vendorCode / supplierArticle;
  * product name;
  * category;
  * active/inactive status;
* нет дублей `wb_nm_id`;
* нет дублей barcode, если они не объяснены;
* есть понятный `sku_match_status`.

Решение:

* Без корректного `SKU_MASTER` нельзя строить надёжную SKU-экономику.
* Все unmatched строки из RAW должны попадать в контроль.

### 17.2. `COST_HISTORY`

Проверить:

* у каждого активного SKU есть себестоимость;
* себестоимость действует на нужный период;
* есть дата начала действия цены;
* нет пропусков по датам;
* нет отрицательной или нулевой себестоимости без объяснения;
* отдельно проверены партии, если себестоимость менялась.

Решение:

* COGS берётся только из `COST_HISTORY`.
* Если COGS отсутствует — SKU нельзя считать полностью закрытым в P&L.

### 17.3. `BUNDLES`

Проверить:

* все наборы EVETIS есть в справочнике;
* каждый набор разложен на компоненты;
* количество компонентов корректное;
* компоненты связаны с `internal_sku`;
* COGS набора считается из COGS компонентов;
* нет набора без состава.

Решение:

* Без корректного `BUNDLES` нельзя считать маржу наборов.
* Наборы не должны считаться как обычный одиночный SKU, если их себестоимость зависит от компонентов.

## 19. Минимальный контрольный период

Для первой проверки использовать небольшой период, чтобы не ломать процесс:

```text
Период проверки: 7 календарных дней
Предпочтительно: последняя полная неделя
```

Если API имеет ограничение на период запроса, период разбивается на допустимые интервалы.

Если API имеет ограничение истории, это фиксируется в разделе соответствующего блока.

## 20. Что считать успешной проверкой блока

Блок можно считать готовым к API-only только если:

1. Точный endpoint и метод подтверждены фактическим запросом.
2. Параметры запроса понятны.
3. Доступная история понятна.
4. Возвращаемые поля зафиксированы.
5. Целевой RAW-лист определён.
6. Дедупликация понятна.
7. Ключи связи с `SKU_MASTER` работают.
8. Unmatched SKU контролируются.
9. P&L-роль показателя определена.
10. Расхождение с Excel/Drive разобрано.
11. Нет двойного учёта.
12. Статус блока обновлён в карте покрытия.
