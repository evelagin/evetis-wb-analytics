# MART — дизайн-документ (v1)

**Проект:** EVETIS WB Analytics
**Дата:** 2026-07-23
**Статус:** ЧЕРНОВИК НА АУДИТ (ChatGPT). Кода нет. Ни один объект в BQ ещё не создаётся.
**Автор черновика:** ассистент. Приёмка — владелец после аудита.

Документ следует правилу проекта «перед кодом: что / зачем / риск / контрольные цифры».
Все числа ниже получены read-only из BQ `wb_raw` 2026-07-23, а не выдуманы.

---

## 0. TL;DR и принятые решения

Строим слой витрин поверх готовых `V_*`-вьюх: **пять фактов** `FACT_ORDERS`,
`FACT_SALES`, `FACT_STOCKS_SNAPSHOT`, `FACT_FINANCE`, `FACT_ADS_DAILY` и **якорь**
`MART_SKU_DAILY` (грейн день × nm_id).

Решения, зафиксированные с владельцем 2026-07-23:
- **Материализация — таблицами** (не вьюхами), пересборка по расписанию.
- **Объём — весь MART целиком** одним дизайном на аудит.

Решения, предлагаемые этим документом (на аудит):
1. Отдельный датасет **`wb_mart`** — не мешать RAW и расчёт (принцип CLAUDE.md).
2. Грейн якоря — **день × nm_id**; ключи `nm_id` + `internal_sku`; несопоставленное —
   явной меткой «не сопоставлено», строки не выкидываем.
3. Финансы — **только** из `V_WB_FINANCE_CANONICAL`, с колонками
   `source_layer` (DAILY/WEEKLY/LEGACY) и `finance_status` (PROVISIONAL/FINAL).
4. Реклама поартикульно — из `V_ADV_CAMPAIGN_STATS` (есть `nmId`), но это
   **директивная атрибуция ~⅔ бюджета**; истина по сумме расхода — `V_ADV_COSTS`.
   MART несёт и атрибутированный расход, и отдельный «нераспределённый» остаток.
5. **PNL (прибыль/маржа) в v1 не считаем** — заблокировано двумя зависимостями
   (нет себестоимости в BQ + не решена база выручки for_pay). MART v1 отдаёт все
   ингредиенты, кроме прибыли. Прибыль — MART v2.
6. Пересборка — **full-refresh `CREATE OR REPLACE`** (объёмы малы), один дневной
   триггер после апстрима, run-log, ScriptLock.

## 1. Что обнаружено (факты из BQ)

### 1.1. Покрытие входов по датам и объёму

| Источник | Вьюха | Даты (min→max) | Строк |
|---|---|---|---|
| Заказы | `V_WB_ORDERS` | 2026-04-13 → 2026-07-23 | 3 719 |
| Продажи/возвраты | `V_WB_SALES_RETURNS` | 2026-03-30 → 2026-07-23 | 3 499 |
| Финансы (канон) | `V_WB_FINANCE_CANONICAL` | **2024-09-05** → 2026-07-22 | 202 057 |
| Реклама (по nm) | `V_ADV_CAMPAIGN_STATS` | 2026-04-13 → 2026-07-22 | 8 489 |
| Остатки (текущие) | `V_WB_STOCKS_CURRENT` | только 2026-07-23 | 148 |

**Ключевой вывод — асимметрия покрытия.** Деньги (финансы) идут с сентября 2024,
а заказы/продажи/реклама — только ~90 дней (WB API хранит 90 дней). Значит в
`MART_SKU_DAILY` за периоды до ~2026-04-13 будут финансовые метрики, но не будет
операционных (заказы/продажи/реклама). Это не дефект — это природа источников;
MART должен строить ось дней объединением дат всех фактов и ставить NULL/0 там,
где источник отсутствует.

`V_WB_STOCKS_CURRENT` отдаёт **только последний снапшот** (148 строк). Для
дневной истории остатков `FACT_STOCKS_SNAPSHOT` берём все COMPLETE-снапшоты из
`WB_STOCKS_SNAPSHOTS` + `RAW_WB_STOCKS` (грейн snapshot × nm_id × warehouse),
а не текущую вьюху.

### 1.2. Сопоставление SKU (sku_match_status)

| Источник | matched | not_found | null | % matched |
|---|---|---|---|---|
| Заказы | 3 715 | 4 | — | 99,9 % |
| Продажи | 3 495 | 4 | — | 99,9 % |
| Финансы | 155 758 | 45 453 | 846 | **77 %** |
| Остатки | 146 | 2 | — | 98,6 % |

Заказы/продажи/остатки сопоставляются почти идеально. У финансов 23 % строк без
SKU — и это **ожидаемо**: значительная часть финансовых строк это не товарные
операции (логистика, хранение, удержания, приёмка, штрафы, эквайринг), у которых
`nm_id` просто нет, плюс исторические nm_id 2024–2025, которых нет в текущем
`REF_SKU_MASTER`. Вывод: `FACT_FINANCE` хранит строки и без nm_id; в
`MART_SKU_DAILY` товарные суммы раскладываем по SKU, а нетоварные (хранение/
удержания без nm) уходят в отдельный «счётный» разрез «не распределено по SKU».
Иначе потеряем часть денег или припишем их не туда.

### 1.3. Реклама: связка SKU → расход (риск CLAUDE.md)

CLAUDE.md прямо запрещает «использовать рекламу WB как точную поартикульную
себестоимость, если источник не даёт связку SKU → расход».

Факт: `V_ADV_COSTS` **не содержит `nmId`** (расход только по `advertId`).
`V_ADV_CAMPAIGN_STATS` содержит `nmId` + `date` + `sum` — то есть поартикульный
расход есть только здесь. Но это fullstats, и он не сходится с costs:

| Окно 30 дней (по 2026-07-23) | Сумма, ₽ |
|---|---|
| `V_ADV_CAMPAIGN_STATS.sum` (по nm) | 103 910 |
| `V_ADV_COSTS.updSum` (истина расхода) | 157 875 |
| Покрытие stats/costs | **65,8 %** |

То есть на этом окне поартикульная атрибуция «видит» лишь ⅔ реального бюджета
(на 90-дневном окне в прошлой сессии было ~93 % — покрытие плавает и, вероятно,
хуже на свежих днях из-за догрузки). **Дизайн-решение:** истина по сумме расхода —
всегда `V_ADV_COSTS`; `campaign_stats` — единственный поартикульный сигнал, но
неполный. `FACT_ADS_DAILY` отдаёт `ad_spend_sku` (по nm, из stats) и отдельно
день-итог из costs; MART несёт `ad_spend_unattributed = costs − Σ stats` как явный
бакет. Поартикульный расход помечаем как директивный, не как точную себестоимость.

Проверка связки nm рекламы со справочником: из 24 уникальных nm в рекламе
23 сопоставлены с `REF_SKU_MASTER`, 1 нет — приемлемо, несопоставленный уйдёт в
«не сопоставлено».

### 1.4. Типы данных

Деньги в `V_WB_SALES_RETURNS` уже `NUMERIC`. А вот в `V_WB_FINANCE_CANONICAL`,
`V_WB_ORDERS`, `V_ADV_*` числовые поля — **STRING** (историческая сырьевая форма) и
могут содержать десятичную запятую. Все факты обязаны нормализовать типы:
`SAFE_CAST(REPLACE(x, ',', '.') AS NUMERIC)`. Ключ nm разного типа между
источниками (STRING в orders/finance/ads, INT в sales/stocks/REF) — приводим к
`INT64` единообразно.

## 2. Целевая модель

Звезда: 5 фактов атомарного/дневного грейна + одна якорная витрина.

### 2.1. FACT_ORDERS
- Источник: `V_WB_ORDERS` (уже дедуп по srid).
- Грейн: 1 строка = 1 заказ (srid).
- Ключевые поля: `order_date`, `nm_id (INT64)`, `internal_sku`, `sku_match_status`,
  `warehouse_name`, `region_name`, `price_with_disc (NUMERIC)`, `quantity (INT64)`,
  `is_cancel`, `cancel_dt`. Нормализация типов из STRING.

### 2.2. FACT_SALES
- Источник: `V_WB_SALES_RETURNS` (дедуп по sale_id).
- Грейн: 1 строка = 1 продажа/возврат (sale_id).
- Поля: `sale_date`, `nm_id`, `internal_sku`, `sku_match_status`, `is_return`,
  `is_realization`, `operation_type`, `total_price`, `price_with_disc`,
  `finished_price`, `for_pay`, `warehouse_name/type`, `region`.
- Продажи и возвраты различаем флагом `is_return` (возврат → отрицательный вклад).

### 2.3. FACT_STOCKS_SNAPSHOT
- Источник: `RAW_WB_STOCKS` × COMPLETE-снапшоты из `WB_STOCKS_SNAPSHOTS`
  (НЕ `V_WB_STOCKS_CURRENT`, та только последняя).
- Грейн: snapshot_date × nm_id × warehouse_id (× chrt_id).
- Поля: `quantity`, `in_way_to_client`, `in_way_from_client`, `internal_sku`.
- Для якоря сворачиваем к день × nm_id (сумма остатка по складам на конец дня).

### 2.4. FACT_FINANCE
- Источник: **только** `V_WB_FINANCE_CANONICAL`.
- Грейн: 1 строка = 1 строка отчёта реализации (rrd_id).
- Обязательно тянем `source_layer`, `finance_status`, `operation_type_normalized`,
  `week_start`, `report_type`. Даты: `rr_dt`/`_rr_date` (основная), `sale_dt`, `order_dt`.
- Деньги (STRING→NUMERIC): `sale_amount`, `return_amount_rub`, `for_pay`,
  `commission_amount`, `logistics_amount`, `storage_fee`, `deduction`, `penalty`,
  `acceptance`, `acquiring_fee`, `additional_payment`, `compensation_amount`.
- Строки без nm_id (нетоварные) сохраняем — они нужны для полноты денег.

### 2.5. FACT_ADS_DAILY
- Источники: `V_ADV_CAMPAIGN_STATS` (поартикульно) + `V_ADV_COSTS` (итог-истина).
- Грейн: date × nm_id × advertId (из stats), плюс дневной контроль из costs.
- Поля: `views`, `clicks`, `orders`, `ad_spend_sku` (из `sum`), `advertType`.
- Инвариант: `Σ ad_spend_sku(date) ≤ costs(date)`; разница = нераспределённый расход.

### 2.6. MART_SKU_DAILY (якорь)
- Грейн: **день × nm_id** (+ атрибуты SKU из `REF_SKU_MASTER` через LEFT JOIN).
- Ось дней = UNION дат всех фактов; несопоставленный nm — «не сопоставлено».
- Метрики (v1): заказы (шт/₽), выкупы (шт/₽), возвраты (шт/₽), рекламный расход
  атрибутированный + нераспределённый остаток по дню, комиссия, логистика,
  хранение, удержания, штрафы, `for_pay`, остаток на конец дня.
- **Прибыль/маржа/ДРР-как-%-от-прибыли в v1 = NULL** (см. §4 блокеры).
  ДРР как расход/выручка — можно (обе части есть), маржа — нет.

## 3. Ключевые дизайн-решения и почему

1. **Таблицы, не вьюхи** — решение владельца: быстрый и дешёвый чтение-доступ для
   будущего дашборда. Цена — отдельный контур эксплуатации (триггер + свежесть).
2. **Датасет `wb_mart`** — не смешивать RAW и расчёт (CLAUDE.md). Обсуждаемо:
   можно оставить в `wb_raw` рядом с `V_*`, но чистое разделение предпочтительнее.
3. **Грейн день × nm_id** — минимальный уровень, на котором сходятся все пять
   источников; внутренний SKU и наборы навешиваются справочником, а не грейном.
4. **«Не сопоставлено» явно** — строки не выбрасываем; иначе теряем деньги/остатки.
5. **Финансы только из canonical со статусами** — свежий день виден как
   PROVISIONAL, закрытая неделя как FINAL; дашборд сможет не смешивать.
6. **Реклама: costs — истина, stats — единственный поартикульный сигнал** —
   прямое следствие §1.3 и запрета CLAUDE.md. Нераспределённый расход — явный бакет.
7. **Full-refresh пересборка** — объёмы малы (крупнейший факт 202k строк),
   `CREATE OR REPLACE` идемпотентен, полностью обратим, «не усложнять». Инкремент
   (MERGE + watermark) вводим только если появится реальная цена/латентность.

## 4. Что заблокировано и почему (границы v1)

- **Себестоимость (COGS) — нет в BQ.** `REF_SKU_MASTER` не содержит цены закупки;
  `COST_HISTORY` живёт в Sheets, перенос — отложенный REF Sync PR2. Без COGS нет
  валовой прибыли и маржи.
- **База выручки for_pay — не решена.** Формула PNL (§8 архитектуры, вариант A/B)
  ждёт reconciliation `for_pay` (продажи `for_pay` vs финансы `for_pay`). Данные для
  сверки теперь копятся сами, но решение о базе ещё не принято.
- **Наборы (bundles).** В `REF_SKU_MASTER` есть `is_bundle`, но разложение набора
  на компоненты в v1 не делаем — nm-набор идёт как отдельный SKU. Флаг на v2.

Вывод: **MART v1 = все ингредиенты прибыли, кроме самой прибыли.** Прибыль/маржа —
отдельный MART v2 после REF PR2 (себестоимость) и решения по for_pay.

## 5. Какие объекты затрагиваются

Создаются **новые** объекты (ничего существующего не меняем):

| Объект | Тип | Датасет |
|---|---|---|
| `FACT_ORDERS`, `FACT_SALES`, `FACT_STOCKS_SNAPSHOT`, `FACT_FINANCE`, `FACT_ADS_DAILY` | TABLE | `wb_mart` |
| `MART_SKU_DAILY` | TABLE | `wb_mart` |
| `MART_LOADER_RUNS` | TABLE (run-log) | `wb_mart` |
| `runWbMartDaily` (+ установщик триггера) | Apps Script `.gs` | repo `apps-script/` |

Существующие `V_*`, `RAW_*`, `REF_*`, триггеры Уровня 1 — **не трогаем**.
Листы Google Sheets — не трогаем (лимит ячеек на 100 %, MART живёт в BQ).

## 6. Материализация и расписание

- Пересборка: `CREATE OR REPLACE TABLE wb_mart.<factt> AS SELECT ... FROM wb_raw.V_*`
  с нормализацией типов. Затем `MART_SKU_DAILY` из фактов.
- Триггер `runWbMartDaily` — **один раз в день ~08:00 МСК**, ПОСЛЕ последнего
  апстрима (финансы/REF ~07:30). Внутридневная свежесть заказов (hourly) в дашборд
  попадёт на следующей пересборке — приемлемо для утреннего управленческого среза.
- ScriptLock + запись в `MART_LOADER_RUNS` (start/finish/rows/status), как у
  остальных загрузчиков. Триггер — только после ручной приёмки и негативных тестов.

## 7. Риски поломки

1. **Двойной учёт возвратов** (в FACT_SALES `is_return` и в FACT_FINANCE
   `return_amount_rub`) → в MART складывать из одного источника на метрику, не из двух.
2. **Приписывание нетоварных денег к SKU** (хранение/удержания без nm) → отдельный
   бакет «не распределено», не размазывать по nm_id.
3. **Реклама как точная себестоимость** → запрещено; несём costs-истину и бакет
   нераспределённого (см. §1.3).
4. **Смешение PROVISIONAL и FINAL финансов** → всегда тянуть статус, дашборд решает.
5. **Тихое устаревание таблиц** (урок OAuth-инцидента: триггеры молча встали) →
   `MART_LOADER_RUNS` + внешний контроль свежести (в бэклоге идей).
6. **Запятая/типы** → единая нормализация STRING→NUMERIC во всех фактах.

## 8. Контрольные цифры для приёмки

После первой сборки проверяем (BQ-SQL):
1. **Дедуп фактов:** rows(FACT_*) == distinct(ключ) — как у `V_*` вьюх.
2. **Реклама-инвариант:** для каждого дня `Σ ad_spend_sku ≤ costs(day)`;
   `ad_spend_unattributed ≥ 0`.
3. **Финансы-полнота:** `Σ FACT_FINANCE.sale_amount` == `Σ V_WB_FINANCE_CANONICAL`
   (тот же фильтр), т.е. факт ничего не потерял; строки без nm сохранены.
4. **Покрытие MART:** min(day) по операционным метрикам ≈ 2026-04-13; финансовые
   метрики уходят к 2024-09 — асимметрия ожидаема, не баг.
5. **Сверка с живой таблицей:** заказы/выкупы за контрольный день из
   `MART_SKU_DAILY` совпадают с дневными числами из книги «Evetis аналитика 2.0».
