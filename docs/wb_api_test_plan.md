## 0. Назначение и границы документа

Документ описывает, **какие тестовые запросы** к WB API нужно выполнить по кабинету EVETIS, **что именно проверить** в ответах и **какое решение** принять по каждому блоку до написания production-кода.

Документ **не**:
- меняет код Apps Script;
- не вводит новых функций;
- не меняет структуру Google Sheets;
- не объявляет WB API полным до фактических тестов;
- не фиксирует контрольные значения как эталон (это делает будущий `docs/control_values.md` после успешных тестов).

### Жёсткие правила (наследуются из политики источников)

- **Excel/Drive** — только legacy / сверка / fallback. В тестах используются **исключительно** для сверки сумм, не как источник факта.
- **Себестоимость (COGS)** в WB API **не ищется** by design (источник — `SKU_MASTER`, `COST_HISTORY`, `BUNDLES`).
- **Реклама не считается SKU-фактом** до успешной сверки `Σ fullstats.sum` ↔ `Σ adv/v1/upd.updSum` ↔ кабинет WB.
- Любое непроверенное значение остаётся `TBD`.

### Контрольный период первого теста

```
2026-05-18 — 2026-05-24  (7 дней)
```

Период выбран как единый для всех блоков. Совместимость с лимитами:
- `paid_storage` (макс. 8 дней) — помещается в одну задачу;
- `acceptance_report` (макс. 31 день) — помещается;
- `adv/v3/fullstats` (макс. 31 день) — помещается;
- Finance / legacy finance — помещается.

### Категории токенов

| Категория | Используется в тестах | Хост |
|---|---|---|
| **Statistics** | T2 (legacy finance), T3 (orders), T4 (sales/returns) | `statistics-api.wildberries.ru` |
| **Finance** | T1 (новый finance) | `finance-api.wildberries.ru` |
| **Analytics** | T5, T6 (остатки), T10 (хранение), T11 (приёмка) | `seller-analytics-api.wildberries.ru` |
| **Promotion** | T7, T8, T9 (реклама) | `advert-api.wildberries.ru` |

### Общие правила выполнения тестов

- Каждый ответ сохраняется **в файл** (raw JSON / выгрузка задачи) с пометкой: endpoint, период, дата/время запроса, категория токена. Сохранение — вне Google Sheets (структура листов не меняется).
- Для task-based методов фиксируются три артефакта: тело `create`, ответ `status`, выгрузка `download`.
- Контрольные суммы считаются **отдельно** (вручную / в черновом расчёте), не записываются в боевые листы до решения.
- Порог сверки денежных сумм наследуется из кода проекта: **±1 ₽** (`FINANCE_RECON_THRESHOLD_`); по рекламе/хранению — отдельные пороги (см. §«Таблица контрольных сумм»).

## 1. Тест T1 — Новый Finance API (целевой финансовый факт)

- **Цель теста.** Подтвердить, что новый Finance API отдаёт детальный отчёт реализации за период с нужной детализацией (операция / дата / `nmId` / строка) и пригоден как **целевой production-источник** финансового факта.
- **Endpoint.** `https://finance-api.wildberries.ru/api/finance/v1/sales-reports/detailed`
  Companion: `…/sales-reports/list` (получить `reportId`), `…/sales-reports/detailed/{reportId}` (детализация по ID).
- **HTTP method.** POST.
- **Token category.** Finance.
- **Request params/body.** Тело запроса:
  - `dateFrom` — начало периода;
  - `dateTo` — конец периода;
  - `limit` — **≤ 100 000**;
  - `rrdId` — курсор пагинации;
  - `fields` — массив для кастомизации набора полей.
- **Пагинация.** Старт — `rrdId = 0`; далее в каждом следующем запросе `rrdId` = идентификатор последней строки предыдущего ответа; повторять **до ответа `204`** (нет данных).
- **Ограничения и лимиты.** Лимит **1 запрос в минуту**. Данные доступны **с 29.01.2024**. Денежные поля приходят **строками (string)**, имена полей — **camelCase**, стандартизованы под другие методы.
- **Что сохранить из ответа.** Полный raw JSON `detailed` за период (все страницы пагинации); ответ `list` (структура `reportId`); финальный `rrdId`; фактические имена полей (для будущей таблицы соответствия camelCase).
- **Какие поля проверить.** Идентификатор строки (аналог `rrd_id`), `nmId`, `srid`/аналог, `vendorCode`, `barcode`, даты (операции/реализации), кол-во, сумма реализации, комиссия, к перечислению, логистика, хранение, удержания/штрафы, приёмка, эквайринг, тип документа/операции. **COGS отсутствует — это ожидаемо.**
- **Контрольные суммы.** `gross_revenue`, `Σ for_pay` (к перечислению), `Σ logistics`, `Σ storage`, `Σ deductions`, `sales_qty`, число строк.
- **Признаки SKU-привязки.** Доля строк с заполненным `nmId`; доля строк без `nmId` (кабинетные/сервисные); корректность маппинга `nmId → internal_sku` по `SKU_MASTER`; доля unmatched `nmId`.
- **Риски.** Смена типа денежных полей (string) → риск парсинга; пагинация по `rrdId` с завершением по `204`; нестабильность набора полей при `fields`; legacy-источник (T2) удаляется 15 июля — на него нельзя опираться долго.
- **Критерий успешности.** Получен полный отчёт за период; идентифицированы все целевые поля; денежные суммы корректно приведены к числу; контрольные суммы сходятся с T2 (legacy) в пределах ±1 ₽ либо расхождение классифицировано (методология / округление); доля unmatched `nmId` зафиксирована.
- **Если тест не прошёл.** Оставить T2 (legacy `reportDetailByPeriod`) как временный источник/сверку до 15 июля; зафиксировать причину; **не закреплять** новый Finance как канон; production-загрузчик финансов **не кодить** (см. §«Нельзя кодить до теста»).

## 2. Тест T2 — Legacy Finance (только сверка)

- **Цель теста.** Получить контрольные суммы реализации из текущего/legacy источника для **сверки с T1**. Роль — сверка, не целевой канон.
- **Endpoint.** `https://statistics-api.wildberries.ru/api/v5/supplier/reportDetailByPeriod`
- **HTTP method.** GET.
- **Token category.** Statistics.
- **Request params/body.** `dateFrom` (RFC3339), `dateTo`, `limit` (≤100 000), `rrdid` (пагинация с 0 по `rrd_id`).
- **Ограничения и лимиты.** **Deprecated, удаление 15 июля.** История с 29.01.2024. Лимит **1 запрос/мин**. При отсутствии данных возвращает `204`.
- **Что сохранить из ответа.** Raw JSON всех страниц за период; финальный `rrd_id` пагинации; число строк.
- **Какие поля проверить.** `rrd_id`, `srid`, `nm_id`, `sa_name`, `barcode`, `ppvz_for_pay`, `delivery_rub`, `storage_fee`, `deduction`, `penalty`, `acceptance`, `acquiring_fee`, `doc_type_name`, `supplier_oper_name` (snake_case — как в текущем `FINANCE_API_FIELD_MAP_`).
- **Контрольные суммы.** Те же, что в T1: `gross_revenue`, `Σ for_pay`, `Σ logistics`, `Σ storage`, `Σ deductions`, `sales_qty`, число строк.
- **Признаки SKU-привязки.** Доля строк с `nm_id`; сопоставление набора `nm_id` с T1 (должны совпадать по периоду).
- **Риски.** Скорое отключение метода; жёсткий лимит 1 запрос/мин (медленная пагинация); `204` при пустом периоде можно принять за ошибку.
- **Критерий успешности.** Суммы T2 получены и используются как опорные для сверки с T1; пагинация по `rrd_id` отработала; пустой период корректно отличён от ошибки.
- **Если тест не прошёл.** Использовать Excel/Drive только как запасную сверку (с пометкой источника); зафиксировать невозможность сверки T1↔T2 и отложить закрепление Finance-канона.

## 3. Тест T3 — Orders (операционная воронка)

- **Цель теста.** Подтвердить операционный источник заказов (воронка, мониторинг). **Не финансовый факт.**
- **Endpoint.** `https://statistics-api.wildberries.ru/api/v1/supplier/orders`
- **HTTP method.** GET.
- **Token category.** Statistics.
- **Request params/body.** `dateFrom` (RFC3339, начало периода), `flag` (0 — инкремент по `lastChangeDate`; 1 — все заказы за дату). Пагинация по `lastChangeDate`.
- **Ограничения и лимиты.** ~80 000 строк/ответ; история ≤ 90 дней; обновление ~каждые 30 мин. Явной пометки Deprecated нет — **мониторить Release Notes**.
- **Что сохранить из ответа.** Raw JSON за период; число строк; `lastChangeDate` последней страницы.
- **Какие поля проверить.** `date`, `lastChangeDate`, `cancelDate`, `supplierArticle`, `nmId`, `barcode`, `subject`, `brand`, `techSize`, `totalPrice`, `finishedPrice`, `priceWithDisc`, `srid`, `gNumber`, `sticker`, `isCancel`.
- **Контрольные суммы.** Число заказов (строк); число уникальных `srid`; `Σ totalPrice`; `Σ finishedPrice`; кол-во `isCancel = true`.
- **Признаки SKU-привязки.** Привязка по `nmId`/`barcode`/`supplierArticle`; доля unmatched `nmId` к `SKU_MASTER`; уникальность `srid` (дедуп).
- **Риски.** Предварительные данные (не факт реализации); 90 дней истории; отменённые заказы; суммы заказов ≠ реализации (нельзя в P&L).
- **Критерий успешности.** Получены строки за период; `srid` уникальны и дедуп работает; доля unmatched зафиксирована; подтверждено, что блок — операционный, не финансовый.
- **Если тест не прошёл.** Понизить статус блока до «требует доработки»; не использовать заказы ни в воронке, ни в P&L до повторного теста; зафиксировать причину (пустой ответ / лимит истории / структура).

## 4. Тест T4 — Sales / Returns (операционные продажи и возвраты)

- **Цель теста.** Подтвердить операционный источник продаж и возвратов. **Не финансовый факт** (факт — из T1).
- **Endpoint.** `https://statistics-api.wildberries.ru/api/v1/supplier/sales`
- **HTTP method.** GET.
- **Token category.** Statistics.
- **Request params/body.** `dateFrom` (RFC3339), `flag` (0/1). Пагинация по `lastChangeDate`.
- **Ограничения и лимиты.** ~80 000 строк/ответ; история ≤ 90 дней. Мониторить Release Notes.
- **Что сохранить из ответа.** Raw JSON за период; число строк; разметку возвратов.
- **Какие поля проверить.** Поля как в orders + `saleID`, `forPay`, `paymentSaleAmount`; признак возврата (`saleID` с отрицательной семантикой / тип операции).
- **Контрольные суммы.** Число продаж; число возвратов; `Σ forPay` (предварительный, не факт); `Σ paymentSaleAmount`.
- **Признаки SKU-привязки.** Привязка по `nmId`/`barcode`; дедуп по `srid`/`saleID`; доля unmatched.
- **Риски.** `forPay` здесь **предварительный** (≠ факт из T1); возвраты в общем потоке; риск двойного учёта выручки, если завести и sales, и finance в P&L.
- **Критерий успешности.** Получены продажи и возвраты за период; возвраты корректно отделимы; дедуп работает; подтверждено, что денежный факт берётся из T1, а не отсюда.
- **Если тест не прошёл.** Понизить статус до «требует доработки»; источник возвратов для факта — Finance-отчёт (T1); зафиксировать причину.

## 5. Тест T5 — Остатки, кандидат A (`warehouse_remains`, task-based)

- **Цель теста.** Проверить task-based сводный отчёт остатков как **кандидат A** на канон остатков (после отключения legacy `supplier/stocks` 23 июня).
- **Endpoint.** `https://seller-analytics-api.wildberries.ru/api/v1/warehouse_remains`
  Поток: `create` → `/tasks/{task_id}/status` → `/tasks/{task_id}/download`.
- **HTTP method.** GET (task-based).
- **Token category.** Analytics.
- **Request params/body.** Группировки (любая комбинация): `groupByBrand`, `groupBySubject`, `groupBySa`, `groupByNm` (при `true` появляется `volume`), `groupByBarcode`, `groupBySize`; фильтры `filterPics`, `filterVolume`; `locale`.
- **Ограничения и лимиты.** Текущий снимок (ретро-история — только через Stocks Report). Отчёт хранится **2 часа**. Время готовности задачи — `TBD`.
- **Что сохранить из ответа.** Тело `create`, ответ `status`, выгрузку `download`; `task_id`; снимок с проставленной `snapshot_date` (на стороне проекта).
- **Какие поля проверить.** `brand`, `subjectName`, `vendorCode`, `nmId`, `barcode`, `techSize`, `volume`, `quantity`, массив `warehouses[]` (разбивка по складам).
- **Контрольные суммы.** `Σ quantity` по всем `nmId`; число уникальных `nmId`; число складов; `Σ quantity` по каждому складу (сходимость с общей суммой).
- **Признаки SKU-привязки.** Привязка по `nmId`/`barcode`/`vendorCode`; один `nmId` на нескольких складах суммируется корректно; доля unmatched.
- **Риски.** Только текущий снимок (историю фиксирует проект через регулярный запуск); 2-часовое хранение отчёта; task-механика → отдельный код загрузчика; конкуренция канона с кандидатом B (T6).
- **Критерий успешности.** Задача создаётся, статус достигает готовности, выгрузка скачивается; поля и детализация (`nmId` × склад) присутствуют; суммы по складам сходятся с общей; unmatched зафиксирован.
- **Если тест не прошёл.** Перейти к оценке кандидата B (T6) как канона; зафиксировать причину (таймаут задачи / неполные поля); **не закреплять** канон остатков; загрузчик остатков не кодить.

## 7. Тест T7 — Реклама: список кампаний

- **Цель теста.** Получить перечень `advertId` кабинета EVETIS для последующего запроса статистики (вход для T8/T9).
- **Endpoint.**
  `https://advert-api.wildberries.ru/adv/v1/promotion/count`
  `https://advert-api.wildberries.ru/adv/v1/promotion/adverts`
- **HTTP method.** GET.
- **Token category.** Promotion.
- **Request params/body.** `count` — без параметров (сводка по статусам/типам); `adverts` — фильтры по статусу/типу кампаний (уточнить по документации). Для статистики важны статусы `7`, `9`, `11`.
- **Ограничения и лимиты.** Лимиты Promotion-категории — соблюдать; набор `advertId` для T8 ограничен **50 на запрос fullstats**.
- **Что сохранить из ответа.** Полный список `advertId` с типом и статусом; сводку `count`.
- **Какие поля проверить.** `advertId`, тип кампании (аукцион/авто/АРК), статус; активность в контрольном периоде.
- **Контрольные суммы.** Число кампаний всего; число кампаний в статусах `7/9/11`; число кампаний, активных в периоде 18–24.05.2026.
- **Признаки SKU-привязки.** На этом шаге SKU-привязки нет (привязка появляется в T8 по `nmId`). Фиксируется только перечень `advertId`.
- **Риски.** Кампании в неподходящих статусах не дают статистику; список нужно разбить на пачки ≤50 для T8.
- **Критерий успешности.** Получен полный список `advertId`; выделены кампании со статистикой за период; список разбит на пачки ≤50.
- **Если тест не прошёл.** Без списка кампаний T8/T9 не выполняются — рекламный блок останавливается до повторного теста; зафиксировать причину.

## 8. Тест T8 — Реклама: fullstats (детализация по `nmId`)

- **Цель теста.** Проверить наличие связки **`date + nmId + advertId + spend`** в статистике кампаний. SKU-фактом считается **только после сверки** в T9 (см. правило).
- **Endpoint.** `https://advert-api.wildberries.ru/adv/v3/fullstats`
- **HTTP method.** GET.
- **Token category.** Promotion.
- **Request params/body.** `ids` (массив `advertId`, **≤ 50**), `beginDate`, `endDate` (контрольный период).
- **Ограничения и лимиты.** Период **≤ 31 день**; **3 запроса/мин**, интервал **20 с**; статистика только для статусов `7/9/11`; синхронизация ~раз в 3 мин. Программное превышение лимита → временная блокировка.
- **Что сохранить из ответа.** Raw JSON по каждой пачке `advertId`; структуру `days[].apps[].nms[]`.
- **Какие поля проверить.** `days[].date`; корневой `advertId`; `days[].apps[].nms[].nmId`; `days[].apps[].nms[].sum`; показы/клики/заказы; типы `appType`.
- **Контрольные суммы.** `Σ sum` (spend) по периоду; `Σ sum` по каждому `advertId`; `Σ sum` по `nmId` (агрегация `sum` по `appType` в рамках дня); `Σ views`, `Σ clicks`, `Σ orders`.
- **Признаки SKU-привязки.** Полнота связки `date + nmId + advertId + spend` (структурно); доля `nmId` вне `SKU_MASTER`; проверка на известный дефект `sum = 0` при наличии заказов.
- **Риски.** Возможен `sum = 0` при наличии заказов (нестабильное поле); типы кампаний разносят расход по-разному; жёсткие лимиты запросов; `sum` — разнесение WB, **не «чистый» прямой расход**.
- **Критерий успешности.** Связка `date + nmId + advertId + spend` присутствует; `Σ sum` посчитан по уровням; дефект `sum=0` проверен; **готово к сверке с T9** (но ещё не SKU-факт).
- **Если тест не прошёл.** Реклама не используется как SKU-факт; spend по `nmId` помечается как **оценка**; зафиксировать причину (пустой `nms[]`, `sum=0`, лимиты).

## 9. Тест T9 — Реклама: списания (`adv/v1/upd`) и сверка

- **Цель теста.** Получить кабинетный итог списаний и выполнить **обязательную сверку** `Σ fullstats.sum` ↔ `Σ upd.updSum` ↔ кабинет WB. Без этой сверки реклама **не SKU-факт**.
- **Endpoint.** `https://advert-api.wildberries.ru/adv/v1/upd`
- **HTTP method.** GET.
- **Token category.** Promotion.
- **Request params/body.** Query-параметры:
  - `from=2026-05-18`
  - `to=2026-05-24`
- **Ограничения и лимиты.** Лимит **1 запрос/сек, burst 5**. История списаний — по документации.
- **Что сохранить из ответа.** Raw JSON списаний за период; суммы `updSum` по дате/кампании.
- **Какие поля проверить.** `updSum` (сумма списания), дата, `advertId`, тип списания/источник средств.
- **Контрольные суммы.** `Σ updSum` за период; `Σ updSum` по `advertId`; **дельта** `Σ fullstats.sum − Σ updSum`; **дельта** `Σ updSum − итог кабинета WB`.
- **Признаки SKU-привязки.** `upd` обычно без `nmId` (кабинетный/кампанийный итог) — SKU-привязка берётся из T8, а **подтверждается** сверкой сумм с T9.
- **Риски.** Расхождение `fullstats.sum` ↔ `upd.updSum`; разные источники средств (счёт/баланс/бонусы); итог кабинета может отличаться от обоих API.
- **Критерий успешности (правило проекта).** Тройная сверка `Σ fullstats.sum ≈ Σ upd.updSum ≈ кабинет WB` в пределах допуска. **Только при сходимости** spend по `nmId` (из T8) можно помечать как **SKU-факт**; иначе — **оценка**.
- **Если тест не прошёл.** Рекламный расход по SKU остаётся **оценкой**; кабинетный итог берётся из `upd`; SKU-факт рекламы **не кодить**; зафиксировать величину и причину расхождения.

## 10. Тест T10 — Хранение (`paid_storage`, task-based)

- **Цель теста.** Проверить task-based отчёт платного хранения как SKU-детализированный источник; сверить с полем хранения финотчёта (T1/T2).
- **Endpoint.** `https://seller-analytics-api.wildberries.ru/api/v1/paid_storage`
  Поток: `create` → `status` → `download`.
- **HTTP method.** GET (task-based).
- **Token category.** Analytics.
- **Request params/body.** `dateFrom`, `dateTo` (контрольный период, 7 дней — помещается).
- **Ограничения и лимиты.** **Макс. период — 8 дней**; отчёт хранится **2 часа**; история — `TBD`.
- **Что сохранить из ответа.** Тело `create`, ответ `status`, выгрузку `download`; `task_id`.
- **Какие поля проверить.** `date`, `nmId`, `barcode`, `vendorCode`, `chrtId`, `warehouse`, `warehousePrice`, `volume`, `calcType`, тарифные даты.
- **Контрольные суммы.** `Σ warehousePrice` за период; `Σ warehousePrice` по `nmId`; сопоставление с `storage_fee` из финотчёта.
- **Признаки SKU-привязки.** `nmId` + `warehousePrice` (SKU-детализация); доля unmatched; в финотчёте хранение часто без `nmId` (агрегат).
- **Риски.** **Двойной учёт хранения** (поле финотчёта + `warehousePrice`); период 8 дней → много задач при больших диапазонах; 2-часовое хранение отчёта.
- **Критерий успешности.** Задача отрабатывает; SKU-детализация хранения получена; `Σ warehousePrice` сверена с финотчётом в пределах допуска (или расхождение классифицировано); выбран **один канон** (рекомендация: `paid_storage` для SKU, финотчёт — сверка).
- **Если тест не прошёл.** Хранение по SKU не закрепляется; использовать агрегат из финотчёта; загрузчик хранения не кодить до выбора канона.

## 11. Тест T11 — Платная приёмка (`acceptance_report`, task-based)

- **Цель теста.** Проверить task-based отчёт платной приёмки; сверить с полем приёмки финотчёта (T1/T2).
- **Endpoint.** `https://seller-analytics-api.wildberries.ru/api/v1/acceptance_report`
  Поток: `create` → `status` → `download`.
- **HTTP method.** GET (task-based).
- **Token category.** Analytics.
- **Request params/body.** `dateFrom`, `dateTo` (`YYYY-MM-DD`, контрольный период).
- **Ограничения и лимиты.** **Макс. период — 31 день**; отчёт хранится **2 часа**.
- **Что сохранить из ответа.** Тело `create`, ответ `status`, выгрузку `download`; `task_id`.
- **Какие поля проверить.** `count`, `giCreateDate`, `incomeId`, `nmID`, `shkCreateDate`, `subjectName`, `total`.
- **Контрольные суммы.** `Σ total` за период; `Σ total` по `incomeId` (поставка); сопоставление с полем `acceptance` финотчёта.
- **Признаки SKU-привязки.** Привязка по `nmID` на уровне поставки (`incomeId`); в финотчёте приёмка чаще партийная/без `nmId`.
- **Риски.** **Двойной учёт приёмки** (финотчёт + acceptance_report); приёмка относится к поставке (`incomeId`), а не к продаже; смешение с удержаниями в общих сервисных строках финотчёта.
- **Критерий успешности.** Задача отрабатывает; детализация по `incomeId`/`nmID` получена; `Σ total` сверена с финотчётом в пределах допуска; выбран **один канон**.
- **Если тест не прошёл.** Приёмка не закрепляется как SKU/партия-факт; использовать агрегат финотчёта; загрузчик приёмки не кодить до выбора канона.

## 15. Блоки, которые нельзя кодить до успешного теста

Production-код **не пишется**, пока соответствующий тест не пройден:

1. **Целевой загрузчик финансов (новый Finance API).** Кодить только после T1: подтверждены body, пагинация по `rrdId` (старт `0`, завершение по `204`), camelCase-таблица полей, парсинг string-сумм, сверка с T2.
2. **Загрузчик остатков.** Кодить только после T5/T6 и **выбора канона** (A или B). Кандидат B (T6) не считать подтверждённым по `docs/openapi` до проверки в Swagger/живом тесте. Legacy `supplier/stocks` не закреплять (отключение 23 июня).
3. **Реклама как SKU-факт.** Кодить только после T8 **и** успешной сверки T9 (`Σ fullstats.sum ≈ Σ upd.updSum ≈ кабинет`). До этого spend по `nmId` — **оценка**.
4. **Загрузчик хранения по SKU.** Кодить только после T10 и выбора единого канона (исключить двойной учёт с финотчётом).
5. **Загрузчик платной приёмки.** Кодить только после T11 и выбора единого канона (исключить двойной учёт и смешение с удержаниями).

Дополнительно (наследуется из политики):
- **Себестоимость (COGS)** в WB API не ищется — источник всегда внутренние справочники.
- **Excel/Drive** в коде остаётся только как legacy / сверка / fallback, не как production-источник.
