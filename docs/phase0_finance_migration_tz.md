# Мини-ТЗ — Фаза 0: миграция финансов на API «Финансы» + корректная раскладка расходов

> Статус: ТЗ на согласование (перед кодом). По правилам CLAUDE.md: что меняем, какие листы, риск, контрольные цифры — потом код.
> Дедлайн: старый `GET /api/v5/supplier/reportDetailByPeriod` отключают **15.07.2026**.

## 1. Что меняем (и зачем)

Две связанные задачи в одном шаге:

**(A) Миграция источника.** Перевести загрузчик финансов с `reportDetailByPeriod` на новые методы категории «Финансы»:
- `POST /api/finance/v1/sales-reports/list` — список доступных отчётов реализации. **Первый шаг — именно он:** определяем ФАКТИЧЕСКУЮ глубину по кабинету.
- `POST /api/finance/v1/sales-reports/detailed/{reportId}` — детализация отчёта.

Глубину **не утверждаем заранее.** По документации детализация нового метода доступна с **01.01.2025**.
- Первый запуск `sales-reports/list` ОБЯЗАТЕЛЬНО выводит фактические периоды всех доступных `reportId` — это и есть проверка глубины.
- Бэкфилл делаем **по всем `reportId`, которые реально вернул `list`** (не по предполагаемым датам).
- **Сентябрь–декабрь 2024:** если его нет в `list` — это ОТДЕЛЬНЫЙ сценарий (Excel/ручной импорт), **НЕ часть Фазы 0**.
- Заказы/реклама глубже 90 дней недоступны в любом случае.

**(B) Корректный RAW с классификацией по статьям.** Каждая операция в `RAW_WB_FINANCE` корректно отнесена к статье (комиссия, эквайринг, логистика, хранение, штрафы, удержания, приёмка/доплаты/компенсации) + техническая категория `UNMAPPED` для неизвестных операций WB. `UNMAPPED` — это НЕ статья P&L, а флаг ошибки классификации (повод дописать маппинг).

**Фаза 0 (термины):** новый источник финансов + сохранение RAW + классификация операций + сверка. **Полноценная раскладка расходов в P&L по день × SKU — Фаза 2.**

## 2. Метрики, которые держим РАЗДЕЛЬНО (определяем здесь, считаем в Фазе 2)

Пять разных величин — не смешивать:
1. **Нативная комиссия WB (КВВ)** — поля `commission_percent` / `commission_amount` (ppvz_vw).
2. **Эквайринг** — `acquiring_fee`. WB выносит отдельной колонкой; «к перечислению» (`for_pay`) уже идёт ЗА его минусом.
3. **Эффективное удержание** — производная метрика `1 − for_pay / retail_amount` (≈37% последние месяцы). Включает КВВ + эквайринг + влияние СПП; это аналитический показатель, НЕ отдельная строка расхода.
4. **Рекламные расходы по SKU** — из `RAW_WB_ADS` (fullstats). Это источник рекламы для P&L.
5. **Удержание «WB Продвижение» из финотчёта** — берём ТОЛЬКО как контроль по кабинету (сверка с `sum(ADS.spend)`), НЕ как расход по SKU и НЕ суммируем с п.4.

**Источник эквайринга:** если эквайринг есть в `sales-reports/detailed` → маппим в `acquiring_fee`. Если детализации недостаточно — отдельный официальный источник `/api/finance/v1/acquiring/*`. В Фазе 0 отдельный acquiring-loader **НЕ внедряем без отдельного согласования**.

## 3. Состав статей расходов в RAW (сборка P&L — Фаза 2)

Поля уже есть в `RAW_WB_FINANCE` (проверено по `COL_MAPPING_`). В Фазе 0 — только корректно их заполняем и классифицируем; формулу прибыли собираем в Фазе 2:

| Статья P&L | Поле RAW (старое → новое camelCase) | Знак |
|---|---|---|
| Выручка (реализация) | `retail_amount` (Продажа − Возврат по `doc_type_name`) | + |
| Комиссия WB (КВВ) | `commission_amount` (ppvz_vw) | − |
| Эквайринг | `acquiring_fee` | − |
| Логистика | `logistics_amount` (delivery_rub) | − |
| Хранение | `storage_fee` | − |
| Штрафы | `penalty` | − |
| Прочие удержания | `deduction` | − |
| Приёмка / доплаты / компенсации | `acceptance`, `additional_payment`, `compensation_amount` | ± |
| К перечислению (контроль) | `for_pay` (ppvz_for_pay) | = |
| Себестоимость (COGS) | из `COST_HISTORY`/`BUNDLES` по SKU/дате | − |
| **Реклама** | из `RAW_WB_ADS` по SKU (НЕ из финотчёта) | − |
| Неизвестная операция | `supplier_oper_name` не сматчился → `UNMAPPED` (флаг, не P&L) | — |

- **Реклама** в финотчёте присутствует только общим удержанием «WB Продвижение» по кабинету — по SKU её берём из `RAW_WB_ADS` (fullstats). Сверяем `sum(ADS.spend)` ↔ удержание «Продвижение» в финотчёте; разницу фиксируем. **Не учитывать дважды.**
- **Хранение** в финотчёте — агрегат за неделю; по SKU/дню детальнее даёт отдельный отчёт платного хранения. Для P&L по SKU помечаем источник хранения и способ разнесения.

Формула прибыли (Фаза 2): Выручка − КВВ − эквайринг − логистика − хранение − штрафы − удержания − COGS − реклама(из `RAW_WB_ADS`). Маржа = Прибыль / Выручка. Удержание «WB Продвижение» из финотчёта в сумму НЕ входит — только сверка.

## 4. Какие листы затрагиваются

- `Wbfinanceloader` (код) — новый источник + маппинг старых имён → camelCase, парсинг сумм-строк.
- `RAW_WB_FINANCE` — структура колонок СОХРАНЯЕТСЯ (новые имена API маппим в существующие колонки, чтобы ничего ниже по цепочке не сломать).
- Новый лист статуса не создаём без отдельного согласования.
- Расчётные `UNIT_SKU_DAILY` / `PNL_TOTAL` — следующая фаза; здесь только готовим корректный RAW.

## 5. Риск поломки

- **Суммы приходят строкой (string)** в новом методе → обязательный парсинг в число (точка/запятая). Невнимательность → поедут цифры.
- **Имена полей сменились на camelCase** + убраны `suppliercontract_code`, `ppvz_supplier_id` → маппить по таблице сопоставления из Журнала изменений WB.
- **Отчёты по reportId, понедельно** → нужна логика list → перебор reportId → detailed с пагинацией; идемпотентность по `row_hash`, replace-slice по периоду (как сейчас).
- **Двойной учёт рекламы** (финотчёт + RAW_WB_ADS) — исключить.
- Старый и новый метод **временно сосуществуют** (до 15.07) — можно сверить параллельно.

## 6. Контрольные цифры (по периодам И группам операций, не одной формулой)

НЕ применяем одну универсальную формулу ко всем строкам. Сверяем группами по `supplier_oper_name` / `doc_type_name`:

- **Сверка методов (новый == старый)** за один общий период, по каждой группе операций — расхождение ≈ 0.
- **Реализация и возвраты** (Продажа/Возврат): сверяем `retail_amount`, `commission_amount`, `acquiring_fee` (при наличии в detailed), `for_pay`.
- **Логистика, хранение, штрафы, удержания, продвижение** — сверяем ОТДЕЛЬНЫМИ группами по `supplier_oper_name`, каждую своей суммой.
- **Итог «К перечислению»** за период бьётся со сводным отчётом WB в кабинете.
- **Реклама (контроль кабинета):** `sum(RAW_WB_ADS.spend)` ↔ группа «WB Продвижение» в финотчёте — фиксируем `delta`, без разнесения по SKU.
- **UNMAPPED:** доля строк/сумм без классификации ≈ 0; любой ненулевой `UNMAPPED` — стоп-сигнал.
- **Sanity:** эффективное удержание `1 − for_pay/retail` ≈ 37% за последние месяцы.
