# data_source_policy.md

> Политика источников данных проекта **EVETIS WB Analytics**.
> Опирается на `docs/current_code_map.md` и `docs/stabilization_plan.md`. Документ учитывает, что `docs/control_values.md` пока не должен фиксироваться как окончательный эталон до проверки API-данных. Документ **фиксирует целевую политику** источников данных и **не меняет код**, не переписывает существующие документы и не утверждает, что WB API уже покрывает все данные.
> Статус: проект на проверку.

---

## 1. Цель документа

- Зафиксировать новое стратегическое решение: целевая архитектура проекта — **API-first / API-only**.
- Определить, что считается производственным источником данных, а что — временным/legacy.
- Задать перечень блоков WB API, покрытие которых нужно проверить до дальнейшей разработки.
- Описать последствия этого решения для уже подготовленных документов (`control_values.md`, `stabilization_plan.md`).

Документ задаёт **целевую** политику. Он не утверждает, что переход уже выполнен, и не объявляет API полным до проверки покрытия (§8–§9).

---

## 2. Новое стратегическое решение

- Целевая архитектура — **API-first / API-only**.
- Производственным источником регулярной аналитики считаются **только данные, полученные через WB API**.
- Excel/Drive-отчёты Wildberries **не должны быть основой регулярного P&L**.
- Excel/Drive допускаются только как временный источник сверки, исторический аудит или fallback при недоступности API.
- Старые функции загрузки Excel/Drive **не удаляются**, а помечаются как legacy/замороженные.
- До дальнейшей разработки нужно проверить покрытие WB API по всем необходимым блокам (§8).

---

## 3. Целевая модель источников данных

```
WB API  →  RAW (производственный слой)  →  CLEAN  →  ЮНИТ / P&L  →  отчётность
                  ↑
Excel/Drive  ─ ─ ┘  (только сверка / исторический аудит / fallback, вне регулярного потока)
```

| Слой | Целевой источник | Примечание |
|---|---|---|
| RAW (регулярный поток) | WB API | Единственный производственный источник для регулярной аналитики |
| RAW (сверка/история/fallback) | Excel/Drive | Только вне регулярного потока, с явной пометкой |
| CLEAN | производный от RAW | Источник не меняется — наследует RAW |
| ЮНИТ / P&L | производный от CLEAN/RAW | Источник факта — RAW из API |

---

## 4. Что считается производственным источником

- Производственный источник — **данные WB API**, загруженные штатными API-загрузчиками в RAW-слой.
- Регулярный P&L, CLEAN и юнит-отчёты в целевой модели строятся только на этих данных.
- Любой регулярный показатель, который пока нельзя получить из API, считается **непокрытым** и фиксируется как пробел (§8), а не закрывается Excel/Drive в обход политики.

---

## 5. Что считается временным/legacy-источником

- Excel/Drive-отчёты Wildberries — **временный/legacy-источник**.
- Допустимые роли (§6): сверка, исторический аудит, fallback при недоступности API.
- Legacy-источник не участвует в регулярном расчёте P&L и не считается источником финансового факта в целевой модели.
- Соответствующие функции остаются в коде с пометкой legacy/заморожены (см. `docs/stabilization_plan.md`).

---

## 6. Роль Excel/Drive-отчётов

| Роль | Допустимость | Условие |
|---|---|---|
| Сверка с API | Допустимо | Только для контроля расхождений, не как основа расчёта |
| Исторический аудит | Допустимо | Для периодов, где API недоступен/неполон |
| Fallback при недоступности API | Допустимо временно | С явной пометкой источника и последующей перезагрузкой из API |
| Основа регулярного P&L | Не допускается | Противоречит целевой политике |
| Источник финансового факта по умолчанию | Не допускается | Факт берётся из API-данных в RAW |

---

## 7. Какие текущие модули попадают в legacy

По карте кода (`docs/current_code_map.md`). Пометить как legacy/заморожены, **не удалять**.

| Модуль / функции | Тип | Роль после перехода |
|---|---|---|
| `Wbfinanceimportfromdrive` (`importWbFinanceReports_2026_05_18_24`, `listWbFinanceReportsInDrive`) | Excel/Drive финансы | legacy: сверка / история / fallback |
| `Wbfinancemonthclose` (`updateWbFinanceMay2026FromDriveOnly`, `previewWbFinanceMay2026DriveFiles`) | Drive-only финансы | legacy: сверка / история / fallback |
| `WbStorageImport` (`importWbStorageReport_2026_05_18_24`) | Excel хранение | legacy: сверка / история / fallback |
| `Wbstoragefolderloader` — Drive-часть папочного загрузчика | Drive хранение | legacy: сверка / fallback (API-часть остаётся производственной) |

Производственными остаются API-загрузчики, в частности `Wbfinanceloader` (`importWbFinanceFromApi`) и API-часть загрузки хранения. Их статус canon уточняется после проверки покрытия (§8–§9).

> Примечание: само разнесение источников в одном RAW уже различается по `load_id` (`FIN_API_` vs `FIN_DRIVE_` / `DRIVE_XLSX_FINANCE_`), что позволяет отделять API-данные от Excel/Drive без изменения кода.

## 9. Как принимать решение, что API полностью заменил Excel

Переход на API-only по конкретному блоку считается состоявшимся только при выполнении всех условий:

- блок данных доступен через WB API в нужной детализации (по дате и nmId/advertId, где применимо);
- данные API за контрольный период сверены с будущим `docs/control_values.md`, который должен быть заполнен только после подтверждения API-данных;
- расхождение API vs Excel/Drive за тот же период разобрано и классифицировано: ошибка загрузки / разница методологии / допустимое округление / непокрытый API-блок;
- API-загрузка идемпотентна и не создаёт дублей (контроль через `IMPORT_LOG_*`, дедуп по `row_hash`);
- решение зафиксировано в `CHANGELOG.md`, а соответствующий Excel/Drive-модуль переведён в legacy.

До выполнения этих условий блок остаётся в смешанном режиме, и Excel/Drive сохраняет роль сверки/fallback (§6).

---

## 10. Что не делать на этапе перехода

| Не делать | Почему |
|---|---|
| Удалять Excel/Drive-функции | Нужны как сверка / история / fallback; помечаются legacy |
| Объявлять API полным до проверки покрытия | Покрытие по блокам ещё не подтверждено (§8) |
| Строить регулярный P&L на Excel/Drive | Противоречит целевой политике |
| Смешивать источники без пометки | Источник каждой строки должен быть различим (`load_id`) |
| Менять ключевые листы без контрольных сумм | По `docs/stabilization_plan.md`; будущий `docs/control_values.md` должен быть заполнен после подтверждения API-данных |
| Реализовывать заглушки API без проверки покрытия | Сначала проверка покрытия, потом разработка |

## 11. Последствия для `control_values.md`

- Документ `control_values.md` пока не должен фиксировать Excel/Drive-цифры как окончательный эталон.
- Период 18–24.05.2026 можно оставить как временный контрольный период для первичной сверки.
- Цифры из Excel/Drive могут использоваться только как предварительный ориентир для сравнения с API.
- Окончательные контрольные значения должны быть подтверждены по API-данным.
- Для каждого блока из §8 нужно отдельно зафиксировать:
  - API-источник;
  - уровень детализации;
  - ключи связки;
  - сумму по API;
  - расхождение с Excel/Drive;
  - решение: API покрывает блок полностью / частично / не покрывает.

## 12. Последствия для `stabilization_plan.md`

- §3 «Канонический финансовый путь»: каноническим остаётся слой `RAW_WB_FINANCE`; в целевой модели его производственное наполнение — **только API**, Excel/Drive — сверка/fallback.
- §6 «Что временно замораживаем» дополняется Excel/Drive-загрузчиками как legacy (§7 настоящего документа).
- Приоритеты стабилизации сохраняются, но получают предусловие: проверка покрытия WB API по блокам (§8) выполняется до развития API-only.
- Правило «один канон» распространяется и на источники: один производственный источник (API) на блок, альтернатива (Excel/Drive) заморожена.

> Настоящий документ не переписывает `stabilization_plan.md`, а фиксирует, какие его пункты затрагивает смена политики источников. Сами правки в тот документ вносятся отдельно, с записью в `CHANGELOG.md`.

## 13. Следующий безопасный шаг

- Провести инвентаризацию покрытия WB API по блокам из §8 (только проверка, без изменения кода).
- Зафиксировать результат покрытия (доступно / частично / недоступно) рядом со статусами §8.
- Для покрытых блоков — снять API-данные за контрольный период и только после проверки зафиксировать их в будущем `control_values.md`.
- Excel/Drive-модули (§7) пометить как legacy/заморожены без удаления.
- Итоги проверки и пометки legacy записать в `CHANGELOG.md`.
