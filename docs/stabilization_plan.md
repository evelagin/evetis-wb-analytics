# Stabilization Plan — EVETIS WB Analytics

> План стабилизации проекта **EVETIS WB Analytics** после первичного аудита кода.
> Опирается на `docs/current_code_map.md`. Документ **не переписывает архитектуру** — он фиксирует безопасный порядок работы до того, как проект пойдёт в развитие.
> Статус: проект на проверку. Версия системы на момент написания: `1.0.3-mvp`.

---

## 0. Стратегическое уточнение по источникам данных

Целевая модель проекта EVETIS WB Analytics — API-first / API-only.

Производственным источником регулярной аналитики должны быть данные, полученные через WB API.

Excel/Drive-отчеты Wildberries не считаются целевым производственным источником. Они могут использоваться только как:
- временная историческая сверка;
- аудит расхождений;
- fallback при недоступности API;
- архивный источник для проверки старых периодов.

Все будущие разработки должны двигаться в сторону полного API-контура:
WB API → RAW → CLEAN → ЮНИТ / PNL → DASHBOARD.

---

## 1. Цель стабилизации

Зафиксировать текущее работающее состояние и убрать риск тихой поломки данных до начала любого развития.

Конкретно стабилизация должна обеспечить:

- один понятный путь для финансов, CLEAN и юнит-экономики (вместо нескольких параллельных);
- защиту боевых данных от тестовых и разрушительных функций;
- обязательные контрольные сверки перед изменением ключевых листов;
- ясность, какие функции «живые», а какие — legacy/заморожены.

Стабилизация **не** включает: новую архитектуру, новые функции, удаление старого кода, доработку заглушек API.

---

## 2. Главный принцип: сначала стабильность, потом развитие

Порядок работы строго последовательный:

1. **Сначала зафиксировать и проверить** то, что уже работает.
2. **Только потом** развивать (API заказов/продаж/рекламы, PNL/дашборд, рефакторинг).

Из этого следуют правила-ограничители на период стабилизации:

| Правило | Формулировка |
|---|---|
| Минимальность | Любое изменение — минимальное, понятное, проверяемое, обратимое (как в `CLAUDE.md`). |
| Без переписывания | Архитектура не меняется, код не переписывается. |
| Без патчей поверх патчей | Новые патчи не накладываются на нерешённые дубли. |
| Один путь | По каждому процессу выбирается один канонический маршрут, остальные замораживаются. |

---

## 3. Канонический финансовый путь

**Решение: `RAW_WB_FINANCE` — главный источник финансового факта WB для текущей модели P&L.**

Каноническим является именно **слой `RAW_WB_FINANCE`**, а не способ его загрузки. API и XLSX — это два способа наполнения одного и того же финансового RAW; они не конкурируют между собой и не создают второй источник факта.

```
WB Statistics API  ┐
                   ├─→  RAW_WB_FINANCE (канонический слой)  →  (CLEAN / юнит)  →  контроль через IMPORT_LOG_FINANCE
XLSX с Drive       ┘
```

| Элемент | Канон | Примечание |
|---|---|---|
| Источник финансового факта | `RAW_WB_FINANCE` | Главный источник финансового факта WB для текущей модели P&L |
| Способ наполнения №1 (API) | `importWbFinanceFromApi` (`Wbfinanceloader`) | Наполняет тот же RAW; дедуп по `row_hash`, идемпотентная очистка своих API-строк |
| Способ наполнения №2 (XLSX) | импорт с Drive (`Wbfinanceimportfromdrive`) | Наполняет тот же RAW; не конфликтует с API-строками (разные префиксы `load_id`) |
| Лог и контроль | `IMPORT_LOG_FINANCE`, `computeFinanceControl_` | Контрольные суммы за период |

На период стабилизации финансовый факт берётся из `RAW_WB_FINANCE` и нигде не подменяется ручными формулами поверх RAW.

---

## 4. Канонический CLEAN-путь

**Решение: `CLEAN_WB_DAILY` остаётся промежуточным слоем. Но до развития нужно выбрать ОДНУ каноническую схему — 15 колонок или расширенную.**

Сейчас в коде сосуществуют две ширины (это зафиксировано в карте кода, §8, §10):

| Схема | Где задаётся | Кто пишет |
|---|---|---|
| 15 колонок (`CLEAN_HEADERS_NEW_`) | `Testrawtoclean` | `resetCleanWbDailyStructure`, тестовый контур |
| Расширенная (`CLEAN_HEADERS_FULL_`) | `Wbfinance` | `buildCleanWbDailyWithFinance` |

Риск: при разной ширине одна функция может срезать колонки данных другой.

Порядок действий на стабилизацию (без изменения кода прямо сейчас):

1. Выбрать одну каноническую схему как стандарт.
2. До выбора — не запускать переключение между схемами на боевой таблице.
3. После выбора — зафиксировать решение в `CHANGELOG.md`, вторую схему пометить как legacy.

До принятия решения CLEAN-путь считается «заморожен по схеме»: пересборка допустима только в рамках уже выбранной ширины с контролем сумм.

---

## 5. Канонический путь юнит-экономики / PNL

**Решение: временно не развивать все три версии юнит-отчёта одновременно.**

Текущее состояние (карта кода, §7): три билдера пишут на листы `ЮНИТ_MM_YYYY`:

| Версия | Функция | Файл | Статус на стабилизацию |
|---|---|---|---|
| v10 (новая) | `buildMonthlyUnitReportV10` | `MonthlyUnitReport_v10` | Предварительный кандидат на канон (хранение по `RAW_WB_STORAGE`, реклама по `RAW_WB_ADS`, встроенные сверки) |
| «с финансами» | `buildMonthlyUnitReportWithFinance` | `Wbfinance` | Заморозить как legacy |
| v1.2 (старая) | `buildMonthlyUnitReport` | `Monthlyunitreport` | Заморозить как legacy |

`buildMonthlyUnitReportV10` рассматривается **только как предварительный кандидат** на каноническую версию. **v10 не утверждена как каноническая до проведения контрольной сверки на выбранном периоде** (см. §8 и §11). До этой сверки v10 не получает статус канона, а две другие версии остаются в коде как legacy.

Принципы:

- Отдельного билдера `PNL_TOTAL`/`PNL_SKU` нет — P&L живёт внутри юнит-отчётов; на стабилизацию это сохраняем как есть.
- Развивается **только одна** версия — после её утверждения. Остальные не дорабатываются, но и не удаляются.
- Любое наполнение `ЮНИТ_MM_YYYY` проходит встроенную сверку CLEAN → ЮНИТ.

Выбор канонической версии фиксируется отдельным решением и записью в `CHANGELOG.md`.

---

## 6. Что временно замораживаем

«Заморозка» = функция остаётся в коде, помечается как legacy, не дорабатывается и не запускается в рабочем процессе.

| Объект заморозки | Где | Причина |
|---|---|---|
| Два неканонических билдера юнита | `Monthlyunitreport`, `Wbfinance` | Оставляем одну версию (§5) |
| Неканоническая схема CLEAN | `Testrawtoclean` или `Wbfinance` (после выбора) | До выбора одной ширины (§4) |
| Дублирующие меню юнита | `addUnitReportMenu` / `addUnitReportMenuV10` | Во избежание запуска не той версии |
| Тестовый контур | `fillTestRawDataForClean`, `fillTestFinanceRawData`, `resetCleanWbDailyStructure` | Не для боевой таблицы (§7, §12) |
| Заглушки API | меню «📡 API WB», «📊 Расчёты» | Это развитие, не стабилизация |

Важно: **до отдельного решения старые функции не удаляются** — только помечаются как legacy/заморожены.

## 7. Опасные функции, которые нельзя запускать без подтверждения

Из карты кода (§11, §13). Запуск только осознанно, на копии или с контрольными суммами до/после.

| Функция | Файл | Эффект |
|---|---|---|
| `resetCleanWbDailyStructure` | `Testrawtoclean` | Полная очистка `CLEAN_WB_DAILY` |
| `fillTestRawDataForClean` | `Testrawtoclean` | Перезапись RAW тестовыми данными |
| `fillTestFinanceRawData` | `Wbfinance` | Перезапись финансового RAW тестовыми расходами |
| `buildCleanWbDailyWithFinance` | `Wbfinance` | Перезапись заголовков + удаление колонок CLEAN |
| `removeAllFinanceApiRows` | `Wbfinancemonthclose` | Удаление API-строк RAW_WB_FINANCE + пересборка CLEAN |
| `clearCleanRowsOutsidePeriod_` / `clearFinanceOwnPeriod_` | `Cleanwbdailyperiod` / `Wbfinanceloader` | Удаление строк за/вне периода |
| `setupWorkbook` / `createAllSheets` | `Code.gs` | Обновление структуры всех листов, удаление лишних колонок |
| `installCostEditTrigger` | `CostManagement` | Включает авто-запись в COST_HISTORY по каждому редактированию |

**Правило:** тестовые функции на боевой таблице не запускаются вообще; разрушительные функции — только с подтверждением и контрольными цифрами.

## 8. Контрольные цифры и сверки

**Решение: `RAW_WB_FINANCE`, `CLEAN_WB_DAILY`, `SKU_MASTER`, `COST_HISTORY` и юнит-отчёты не меняются без контрольных сумм.**

Опираемся на уже существующие механизмы (карта кода, §15) — новые писать не нужно:

| Механизм | Что сверяет |
|---|---|
| `IMPORT_LOG_FINANCE` | Итоги загрузки финансов: строки, реализация, к перечислению, логистика, хранение, удержания, «после расходов WB» |
| `IMPORT_LOG_STORAGE` | Итоги загрузки хранения |
| `computeFinanceControl_` | Контроль сумм за период из RAW_WB_FINANCE |
| Сверка CLEAN → ЮНИТ | Встроена в билдер юнита (расхождения → `ERRORS_CONTROL`) |
| `verifyStorageReconciliation` | Хранение: RAW_WB_STORAGE vs RAW_WB_FINANCE по пересечению дат |
| `writeReconciliationErrors_` | ADS vs «WB Продвижение» из финотчёта |
| `auditSkuCostLinks`, `auditSkuAndCost` | Связка SKU_MASTER ↔ COST_HISTORY, дубли `is_current` |
| `ERRORS_CONTROL` | Общий журнал расхождений |

Минимальный обязательный набор контрольных цифр перед любым изменением ключевого листа: продажи (шт), реализация WB, к перечислению WB, логистика, хранение, удержания, «после расходов WB», себестоимость продаж, прибыль.

## 9. Первый приоритет стабилизации

Зафиксировать и проверить финансовый факт.

| Шаг | Действие |
|---|---|
| 1.1 | Подтвердить, что `RAW_WB_FINANCE` — единственный источник финансового факта; ручных формул поверх RAW нет. |
| 1.2 | Снять контрольные суммы по `RAW_WB_FINANCE` за опорный период 18–24.05.2026 или другой период, который владелец проекта подтвердит как контрольный, и зафиксировать их как эталон. |
| 1.3 | Проверить, что API- и XLSX-загрузки не дублируют строки (разные `load_id`, дедуп по `row_hash`). |
| 1.4 | Убедиться, что опасные/тестовые функции не привязаны к рабочему меню без подтверждения. |

Выбрать канонический CLEAN и зафиксировать его.

| Шаг | Действие |

Выбрать одну версию юнит-экономики / PNL.

| Шаг | Действие |
|---|---|
| 3.1 | Провести контрольную сверку `buildMonthlyUnitReportV10` на выбранном периоде; только при успешной сверке утвердить её как каноническую. |
| 3.2 | Две другие версии пометить как legacy/заморожены, оставить в коде. |

## 12. Что пока не делать

| Не делать | Почему |
|---|---|
| Запускать тестовые функции на боевой таблице | Затирают RAW / CLEAN |
| Удалять старые функции | До отдельного решения только пометка legacy |
| Развивать все три версии юнита одновременно | Только одна каноническая (§5) |
| Менять структуру колонок ключевых листов без сверки | Нужны контрольные суммы (§8) |
| Реализовывать заглушки API (заказы/продажи/реклама/остатки) | Это развитие, не стабилизация |
| Распределять рекламу по SKU как точный факт | Только при надёжной связке `SKU → расход` из источника; иначе с пометкой «оценка» |
| Накладывать новые патчи поверх нерешённых дублей | Усложняет стабилизацию |

---

## 13. Правила для следующих изменений

| Правило | Формулировка |
|---|---|
| Перед кодом — описание | Сначала: что меняется, какие листы затронуты, риск поломки, контрольные цифры, и только потом код (по `CLAUDE.md`). |
| Один канон | По финансам, CLEAN и юниту — один путь; альтернативы заморожены. |
| Контрольные суммы обязательны | `RAW_WB_FINANCE`, `CLEAN_WB_DAILY`, `SKU_MASTER`, `COST_HISTORY`, юнит-отчёты — только со сверкой. |
| Legacy вместо удаления | Старые функции помечаются, не удаляются, без отдельного решения. |
| Реклама по SKU — не факт без связки | Только при надёжной связке `SKU → расход`, иначе помечать как «оценка». |
| Запись в `CHANGELOG.md` | Каждое изменение: дата, что, зачем, какие листы, как проверить. |
| Обратимость | Любое изменение должно быть откатываемым. |

---

*Документ фиксирует порядок стабилизации и не вносит изменений в код. Канонические выборы (схема CLEAN, версия юнита) принимаются отдельными решениями с записью в `CHANGELOG.md`.*
