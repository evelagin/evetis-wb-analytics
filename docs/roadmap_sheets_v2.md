# Roadmap — EVETIS WB Analytics на Google Sheets (v2)

> Решение от 28.06.2026: остаёмся на Google Sheets (без сервера, без доп. затрат). Цель — довести скелет до ума: данные сами затягиваются по триггерам, корректно обрабатываются, сверху наглядный дашборд с выбором любой даты/диапазона и полной раскладкой расходов по кабинету и по каждому SKU. Роли доступа пока НЕ внедряем.

## Целевая картина (что хотим на выходе)
- Зашёл утром — данные за вчера уже подтянулись сами (триггеры), кнопки жать не нужно.
- Вижу за любой день / произвольный диапазон (от старта продаж до сегодня): заказы, выкупы, возвраты, расход на рекламу, логистику, хранение, приёмку, комиссию, прочие удержания, штрафы — **каждый расход назван, ничего в «прочее»**.
- Два разреза: **по кабинету целиком** и **по каждому SKU отдельно**.
- Прибыль и маржинальность с учётом себестоимости; ДРР.
- Визуал — веб-страница (Apps Script web-app): выпадающий выбор SKU, выбор дат (пресеты 7/14/30 + произвольный диапазон + конкретный день), графики по дням. Не «бесконечные столбцы вправо».

## Архитектурный принцип (остаёмся в Sheets)
- **Данные и логика** — в Google Sheets (RAW → CLEAN/UNIT → P&L).
- **Визуал** — HTML web-app через Apps Script, читает агрегаты из листов. Прообраз уже есть: `ads-dashboard.html`.
- Один источник правды на показатель; RAW и расчёт не смешиваем (правило из CLAUDE.md).

## ФАЗА 0 — СРОЧНО до 15.07.2026: миграция финансов
**Проблема:** WB отключает `GET /api/v5/supplier/reportDetailByPeriod` 15 июля 2026. На нём держится загрузчик финансов (`Wbfinanceloader` → `RAW_WB_FINANCE`) — источник ВСЕХ удержаний и налоговой базы. После отключения финансы встанут.

**Что делаем:** переводим загрузчик на категорию «Финансы»:
- `POST /api/finance/v1/sales-reports/list` — список отчётов реализации;
- `POST /api/finance/v1/sales-reports/detailed/{reportId}` — детализация.
Нюансы нового формата: имена полей camelCase; суммы приходят строкой (string) — парсить в число; новое поле `title`; выбор полей через массив `fields`. Сопоставление старых/новых имён — в Журнале изменений WB API.

**Затрагивается:** `Wbfinanceloader`, маппинг колонок `RAW_WB_FINANCE`, токен (нужна категория «Финансы», персональный/сервисный).
**Риск:** другой формат сумм (string) и имён — при невнимательном маппинге поедут цифры.
**Контроль:** сумма «к перечислению» и удержаний за один и тот же период старым и новым методом должна совпасть (пока старый ещё жив — можно сверить параллельно).

## ФАЗА 1 — Надёжная ежедневная загрузка (триггеры)
**Цель:** каждый блок тянется сам, идемпотентно, без дублей.
1. Проверить/настроить дневной триггер (`WbDailyRefresh`): заказы, продажи/возвраты, финансы (новый метод), реклама (fullstats), хранение, остатки, приёмка.
2. Лист-индикатор «свежести»: по каждому источнику — дата последней загрузки, число строк, статус OK/ошибка. Чтобы сразу видеть, если что-то не подтянулось.
3. Бэкфилл истории «назад» настолько, насколько отдаёт API (к старту продаж): заказы/продажи/финансы/реклама помесячно. Проверить эмпирически глубину orders/sales (прогнать январь). Остатки исторически невосстановимы — копим только вперёд.

**Затрагивается:** RAW_*-листы, триггеры, новый лист статуса загрузки.
**Риск:** лимиты WB (80k строк/ответ, ~1 req/min), 6-мин лимит Apps Script — грузить помесячно.
**Контроль:** min/max даты и помесячные суммы по каждому RAW сверить с кабинетом WB (как делали за апр–июнь, сходилось 1–2%).

## ФАЗА 2 — Корректная обработка (единый дневной факт)
**Цель:** один чистый слой «день × SKU» со всеми расходами, названными явно, → P&L.
1. Развязать `CLEAN_WB_DAILY`: сейчас мешает API-заказы и Excel-реализацию (даёт некорректную «выручку»). Источник — только API RAW.
2. `UNIT_SKU_DAILY`: на каждый день/SKU — заказы, выкупы, возвраты, реклама, логистика, хранение, приёмка, комиссия, удержания, COGS (из `SKU_MASTER`/`COST_HISTORY`/`BUNDLES`).
3. Расходы без привязки к SKU (кабинетные удержания, приёмка) — отдельным блоком, не размазывать по SKU вручную (правило карты покрытия).
4. Заполнить `PNL_TOTAL` с контрольными суммами: выручка − все расходы − COGS = прибыль; маржа; ДРР. Согласовать единый знаменатель ДРР.

**Затрагивается:** `CLEAN_WB_DAILY`, `UNIT_SKU_DAILY`, `PNL_TOTAL`.
**Риск:** двойной учёт расхода; рассинхрон схемы и факта (живые листы — источник правды).
**Контроль:** сумма расходов в P&L = сумма удержаний в `RAW_WB_FINANCE`; прибыль за период бьётся с выпиской WB.

## ФАЗА 3 — Дашборд (web-app, главный визуал)
**Цель:** то самое удобное «лицо» в браузере.
1. Серверная часть (Apps Script): функции, отдающие агрегаты из `UNIT_SKU_DAILY`/`PNL_TOTAL` по выбранному периоду и SKU.
2. HTML-дашборд (по образцу `ads-dashboard.html`): выбор периода (день / 7 / 14 / 30 / произвольный диапазон от старта продаж), выпадающий список SKU, KPI кабинета, таблица «расходы по SKU», графики по дням (выручка, расходы по статьям, прибыль, ДРР), drill-in по SKU.
3. Публикация как web-app (доступ по ссылке), читает живые данные после дневного триггера.

**Затрагивается:** новый Apps Script web-app + HTML; читает существующие листы (новые листы не создаём без согласования).
**Риск:** объём данных за всю историю — агрегировать на стороне Apps Script, не тащить всё в браузер.
**Контроль:** цифры на дашборде = цифры в `PNL_TOTAL`/`UNIT_SKU_DAILY` за тот же период.

## ФАЗА 4 — Доводка (после основного)
- Оборачиваемость и план поставок по складам; ABC/XYZ.
- Воронка (показы/корзина/CTR/конверсии) — маркетинговый слой.
- Налоговая база (УСН) по юрлицу: расчёт от реализации/к перечислению.
