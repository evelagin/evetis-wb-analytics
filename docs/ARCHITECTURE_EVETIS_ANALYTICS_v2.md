# Архитектура EVETIS Analytics v2

> Единый справочник модели данных. Все будущие загрузчики, витрины и экраны
> **реализуют** эту архитектуру, а не определяют её по ходу. Меняем документ —
> потом код, не наоборот.
>
> Зафиксировано: 2026-07-16.
> **Ревизия 2** — уточнён статус рекламы/финансов, введена двухуровневая модель
> товара (`nm_id` ↔ `internal_sku`), история остатков вынесена в отдельный FACT,
> введена корректная терминология атрибуции, зафиксированы разные денежные базы.
> **Ревизия 3** — формула прибыли/маржи снята с фиксации до reconciliation
> (`for_pay` уже нетто → был риск двойного вычитания удержаний); единое правило
> получения `internal_sku` через `LEFT JOIN REF_SKU_MASTER` с обработкой
> неизвестного `nm_id`; варианты карточки вынесены в `DIM_LISTING_VARIANT`.

## 0. Что мы строим

Не набор загрузчиков и не «отчёты по Wildberries», а **собственную BI-систему
управления бизнесом EVETIS** (свой аналог Power BI/Looker под наш бизнес).
Цель: утром открыл одну страницу — за минуты понял, что происходит с продажами,
рекламой, запасами и финансами, и что нужно сделать сегодня.

Система делится на **четыре продуктовых домена** (§11): Продажи, Маркетинг,
Финансы, Операции. Каждый — законченный модуль со своими витринами и дашбордом,
но все стоят на одной модели данных.

---

## 1. Принципы

- **BigQuery-first.** Google Sheets — только «пульт»: справочники, настройки,
  API-ключи, журналы. Не хранилище и не расчётный слой.
- **Поток данных:** `WB API → Apps Script (только загрузчики/оркестрация) →
  RAW (append-only, неизменяемая история) → VIEW (дедуп/каноническое состояние) →
  REF (справочники из Sheets) → MART (готовые бизнес-показатели) → Dashboard`.
- **Conformed dimensions** — единые ключи во всех витринах. Без них drill-down
  «клик по товару → продажи/реклама/остатки/финансы» невозможен (§3).
- **Fail-closed загрузчики**, каноничность — во VIEW, снапшоты — через manifest.
- Не усложнять; изменения минимальны/обратимы/проверяемы; **PNL без контрольных
  сумм не трогаем**; рекламу не считаем поартикульной себестоимостью без связки
  `SKU→расход`; всё пользовательское — по-русски.

## 3. Единые ключи (conformed dimensions) — сердце модели

### 3.1. Товар — ДВА уровня, а не один

Ключевое архитектурное решение. Нельзя сводить всё к `internal_sku`: реклама,
остатки, позиции и кампании работают именно с **карточкой маркетплейса**.

```
nm_id          — конкретная карточка Wildberries (листинг)
internal_sku   — внутренний товар EVETIS
```

| DIM | Ключ | Смысл |
|---|---|---|
| `DIM_LISTING` | `marketplace` + `nm_id` | карточка маркетплейса → `internal_sku` |
| `DIM_PRODUCT` | `internal_sku` | внутренний товар EVETIS |

**Правило: во ВСЕХ FACT-таблицах храним ОБА ключа — `nm_id` и `internal_sku`.**
Иначе при нескольких карточках одного продукта, новых карточках, наборах или
переносах рекламы аналитика начнёт смешивать разные сущности.

- Агрегируем по `internal_sku` — прибыль, себестоимость, общая аналитика товара.
- Работаем по `nm_id` — реклама, остатки, позиции, кампании, карточка.
- `marketplace` заложен на будущее (планируется Ozon); сейчас всегда `WB`.

**Наборы (bundles)** — различаем два разных SKU:

```
sellable_sku   — то, что реально продано (набор)
component_sku  — компоненты, по которым раскладывается себестоимость
```

**Выручка остаётся на проданном наборе (`sellable_sku`), себестоимость
раскладывается по компонентам** через `REF_BUNDLES`.

### 3.2. Дата

Бизнес-день по Москве. Каждый факт приводит свою дату к дню: продажи `sale_dt`,
заказы `order_dt`, реклама `date`, остатки `snapshot_date`, финансы `rr_date`.

### 3.3. Склад

⚠️ Трещина, которую надо закрыть: у остатков ключ — `warehouse_id` (INT,
стабильный) + `warehouse_name` + `region`; у продаж склад — только
`warehouse_name` (строка, ~23 имени), `warehouseId` **нет**. Мост
`sales_name → warehouse_id` живёт в `REF_WAREHOUSES`; несматченные имена
заносим вручную.

### 3.4. Кампания

`advertId` → тип/имя (`DIM_CAMPAIGN`).

### 3.5. Природа факта во времени

*Событие* (заказы/продажи/финансы/реклама-день) против *снимка* (остатки —
единственный snapshot-факт).

## 3A. Денежные базы — не смешивать

Слово «выручка» слишком общее. Фиксируем разные базы; на одном экране они
**не смешиваются**, каждая метрика явно указывает свою базу.

| База | Что это | Источник | Где применяется |
|---|---|---|---|
| `order_amount` | сумма заказов | `V_WB_ORDERS` | воронка заказов |
| `sale_amount` | сумма состоявшихся продаж | `V_WB_SALES_RETURNS` | продажи, операционка |
| `for_pay` | база к перечислению продавцу — ⚠️ **уже НЕТТО** после удержаний WB | продажи / финотчёт | PNL (осторожно, см. §8) |
| `ad_attributed_revenue` | выручка, которую **рекламный API приписал рекламе** | рекламный API | ROAS/ДРР рекламы |
| `finance_revenue` | сумма по финансовому отчёту | `V_WB_FINANCE` | PNL, факт денег |

Основная операционная воронка: `Заказы → order_amount`, `Продажи → sale_amount`,
`Реклама → ad_attributed_orders / ad_attributed_revenue`, `PNL → финотчёт и for_pay`.

## 3B. Атрибуция рекламы — корректная терминология

**Нельзя** считать `общие заказы − рекламные заказы = органические заказы` без
доказанного контракта сопоставления. Рекламный API учитывает заказ по своему
**окну атрибуции**, а основной API — по дате создания заказа: один и тот же заказ
может попасть в разные дни.

Используем термины:

```
attributed_ad_orders    — заказы, атрибутированные рекламой WB
non_attributed_orders   — остаток после сопоставления
```

Называть `non_attributed_orders` «органикой» можно **только после доказанного
контракта сопоставления** (probe + сверка окон атрибуции). До этого — не называем.

## 4. REF — справочники (Sheets → BQ)

Отдельный небольшой модуль REF-sync (идемпотентный, по расписанию/кнопке).
Пока справочники живут в «пульте» — до этого слоя MART не посчитает маржу/PNL.

| Таблица | Ключ | Грейн | Источник (лист) | Назначение |
|---|---|---|---|---|
| `REF_SKU_MASTER` | `marketplace`+`nm_id` → `internal_sku` | 1 строка = карточка | `SKU_MASTER` | питает `DIM_LISTING`/`DIM_PRODUCT` |
| `REF_COST_HISTORY` | `internal_sku` + дата | дата-эффективная | `COST_HISTORY` | себестоимость по датам |

| `REF_WAREHOUSES` | `warehouse_id` | 1 строка = склад | новый / из остатков | мост `sales_name↔warehouse_id`, регион |
| `REF_SUPPLIERS` | — | — | — | опционально |

## 6. FACT-слой (проектируемое)

Факты — тонкие вью/витрины над VIEW+REF, приведённые к conformed-ключам.

### Правило получения `internal_sku` (обязательное для ВСЕХ conformed FACT)

Каждый conformed FACT несёт **оба** ключа: `nm_id` (из источника) и
`internal_sku` (из справочника). Механика едина и обязательна:

```sql
FACT_*  LEFT JOIN REF_SKU_MASTER  USING (marketplace, nm_id)
```

- **Только `LEFT JOIN`, никогда `INNER`** — факт не имеет права терять строки
  из-за пробела в справочнике.
- **Неизвестный `nm_id`:** `internal_sku IS NULL`, `sku_match_status='not_found'`.
  Строка сохраняется. (Это уже работающее поведение загрузчика остатков; живой
  пример — `nm_id 1083392113`.)
- **Витрины, агрегирующие по `internal_sku`, обязаны явно обрабатывать NULL** —
  выделять отдельным блоком «не сопоставлено», а не молча терять. Молчаливая
  потеря строк в MART считается дефектом.

Это правило применяется ко всем фактам ниже, включая `FACT_ORDERS`,
`FACT_STOCKS_SNAPSHOT` и `FACT_ADS_DAILY` (у них в источнике только `nm_id`).

| FACT | Грейн | Источник | Ключевые меры | Статус |
|---|---|---|---|---|
| `FACT_SALES` | day × nm_id × warehouse_id | `V_WB_SALES_RETURNS` + `REF_WAREHOUSES` + **LEFT JOIN `REF_SKU_MASTER`** | продано шт, `sale_amount`, `for_pay`, возвраты | ⬜ |
| `FACT_ORDERS` | day × nm_id | `V_WB_ORDERS` + **LEFT JOIN `REF_SKU_MASTER`** | заказы шт, `order_amount` | ⬜ |
| **`FACT_STOCKS_SNAPSHOT`** | snapshot_date × nm_id × chrt_id × warehouse_id | **`RAW_WB_STOCKS` JOIN manifest `WHERE status='COMPLETE'`** + **LEFT JOIN `REF_SKU_MASTER`** | остаток, в пути к/от клиента | ⬜ |
| `FACT_FINANCE` | day × nm_id | `V_WB_FINANCE` + **LEFT JOIN `REF_SKU_MASTER`** | комиссия, логистика, хранение, эквайринг, штрафы, `finance_revenue`, `for_pay` | ⬜ |
| `FACT_ADS_DAILY` | day × campaign × nm_id | `V_ADV_*` fullstats + **LEFT JOIN `REF_SKU_MASTER`** | расход, показы, клики, `attributed_ad_orders`, `ad_attributed_revenue` | ⬜ |
| `FACT_ADS_KEYWORDS` 🟡 | day × campaign × keyword (× nm_id?) | **probe нужен** | показы, CTR, CPC, расход, заказы | ⬜ |
| `FACT_ADS_BIDS` 🟡 | запрос × ставка | **probe нужен** | ставка, расход, показы, позиция | ⬜ |

**Важно по остаткам:** исторический факт строится по **всем подтверждённым
(COMPLETE) снимкам** из `RAW_WB_STOCKS`, а не из `V_WB_STOCKS_CURRENT`. Только так
считаются «остаток вчера», динамика запасов, оборачиваемость на дату и анализ
поставок. `V_WB_STOCKS_CURRENT` используется исключительно для текущего состояния.

🟡 = связку `запрос/ставка → расход → SKU` сначала прощупываем probe. Если API её
не даёт — витрину не строим (правило CLAUDE.md).

## 7. DIM-слой

| DIM | Ключ | Атрибуты |
|---|---|---|
| `DIM_LISTING` | `marketplace` + `nm_id` | `internal_sku`, `vendorCode`, категория/предмет/бренд карточки, статус — **только атрибуты уровня карточки** |
| `DIM_LISTING_VARIANT` | `marketplace` + `nm_id` + `chrt_id` | `barcode`, `techSize` — **вариант (размер/баркод) карточки** |
| `DIM_PRODUCT` | `internal_sku` | наименование, категория/бренд, признак набора (`is_bundle`) |
| `DIM_WAREHOUSE` | `warehouse_id` | `warehouse_name`, `sales_name`, `region`, признак агрегатного склада |
| `DIM_DATE` | `date` | день/неделя/месяц/год, день недели, флаг выходного |
| `DIM_CAMPAIGN` | `advertId` | тип (авто/поиск/каталог/…), имя, статус |

⚠️ **Почему варианты вынесены отдельно:** у одной карточки (`nm_id`) может быть
несколько `chrt_id`/`barcode` (размеры/варианты). Держать их скалярными
атрибутами при ключе `marketplace+nm_id` нельзя — получится «схлопывание» и
неверные джойны. `FACT_STOCKS_SNAPSHOT` работает на уровне варианта (`chrt_id`),
продажи несут `barcode`/`techSize` — оба ложатся на `DIM_LISTING_VARIANT`.

## 8. MART — каталог витрин

**Определения метрик (фиксируем один раз, с явной денежной базой):**
CTR = клики/показы; CPC = расход/клики; CPM = расход/показы×1000;
CR = `attributed_ad_orders`/клики;
**ROAS (рекламный)** = `ad_attributed_revenue` / расход;
**ДРР (общий)** = расход / `sale_amount` — доля рекламы в общей выручке;
ACOS = расход / `ad_attributed_revenue` (в %);
дней запаса = остаток / средние продажи в день.
На экране всегда подписываем, какая база используется.

### ⚠️ Формула прибыли/маржи — TBD до reconciliation (не фиксируем)

**Риск двойного вычитания.** `for_pay` — это сумма **к перечислению продавцу**,
то есть она **уже включает вычет** комиссии, логистики и части удержаний WB.
Поэтому формула вида `for_pay − cogs − комиссия − логистика − хранение −
эквайринг − реклама` **списала бы расходы WB дважды** и занизила прибыль.

Пока фактическая семантика полей WB не сверена, формулу **не закрепляем**.
Зафиксировать нужно **ровно один** из двух контрактов:

- **Вариант A (от валовой базы):** берём валовую финансовую базу (продажная
  стоимость по финотчёту) и вычитаем из неё **все** удержания по отдельности:
  `валовая − комиссия − логистика − хранение − эквайринг − прочие удержания −
  cogs − реклама`.
- **Вариант B (от `for_pay`):** берём `for_pay` и вычитаем **только то, что в
  него точно НЕ входит**: `for_pay − cogs − реклама − (удержания вне for_pay)`.

Выбор делаем **после reconciliation**: разложить реальные строки `V_WB_FINANCE`
и доказать на контрольных суммах, какие именно удержания уже сидят в `for_pay`,
а какие идут отдельными строками. До этого `MART_PNL_*` не строим и в дашборд
прибыль не выводим (правило: `PNL_TOTAL` без контрольных сумм не трогаем).

## 12. Открытые вопросы, зависимости, риски

- **Атрибуция рекламы** — доказать контракт сопоставления (окно атрибуции
  рекламного API vs дата создания заказа) прежде чем называть остаток «органикой».
- **Реклама по запросам/ставкам** — probe до проектирования: даёт ли API связку
  `запрос/ставка → расход → SKU`.
- **Реклама vs финансы** — `fullstats` = управленческий поартикульный расход;
  финотчёт = недельная сумма списаний (НЕ поартикульно). Не смешивать.
- **Склад** — сверить ~23 имени продаж с `warehouse_id` остатков, завести
  `REF_WAREHOUSES`, решить судьбу несматченных.
- **Наборы** — раскладка `sellable_sku → component_sku` для cogs; выручка на наборе.
- **Себестоимость** — `REF_COST_HISTORY` дата-эффективная.
- **🔴 Семантика `for_pay` и состав удержаний — reconciliation ДО фиксации формулы
  PNL.** Разложить реальные строки `V_WB_FINANCE` и доказать контрольными суммами,
  какие удержания уже включены в `for_pay`, а какие идут отдельно. Выбрать
  вариант A или B (§8). До этого `MART_PNL_*` не строим и прибыль на дашборд не выводим.
- **Варианты карточки** — подтвердить, что `chrt_id`↔`barcode` соответствие
  устойчиво, и наполнить `DIM_LISTING_VARIANT` (у продаж есть `barcode`/`techSize`,
  у остатков — `chrt_id`).
- **Несколько карточек одного товара** — `DIM_LISTING` должен корректно
  схлопывать несколько `nm_id` в один `internal_sku`.
- **Данные** — 1 unmatched `nm_id` в остатках (`1083392113`) внести в `SKU_MASTER`.
- **Ёмкость Google Sheets** на пределе — RAW из Sheets убирать после аудита.

## 14. Инварианты (неизменны)

RAW append-only; VIEW каноничен; снапшоты — через manifest + статус COMPLETE
(история строится только по COMPLETE); загрузчики fail-closed.

Conformed-ключи `nm_id` **и** `internal_sku` обязательны во всех conformed FACT,
причём `internal_sku` получается **только** через `LEFT JOIN REF_SKU_MASTER`
(никогда INNER); неизвестный `nm_id` → `internal_sku IS NULL` +
`sku_match_status='not_found'`, строка сохраняется; витрины обязаны явно
показывать «не сопоставлено», молчаливая потеря строк — дефект.

Атрибуты уровня карточки и уровня варианта не смешиваются (`DIM_LISTING` vs
`DIM_LISTING_VARIANT`). Денежные базы не смешиваются; **формула прибыли/маржи
не закрепляется до reconciliation `for_pay`** (риск двойного вычитания удержаний);
нет поартикульной рекламной себестоимости без связки `SKU→расход`; «органика» —
только после доказанной атрибуции; `PNL_TOTAL` без контрольных сумм не трогаем;
всё пользовательское — по-русски.
